Chapter 00

한 줄 요약 & 이전 복습

G1의 Concurrent Marking이 안전한 이유 — SATB + Pre-Write Barrier + TAMS 세 장치가 핵심.

"G1의 Concurrent Marking이 안전한 이유는 SATB(Snapshot At The Beginning) 알고리즘이다. Pre-Write Barrier가 참조 변경 직전에 기존 참조 객체를 SATB 버퍼에 푸시해 일시적으로 살리고, Concurrent Start 시점의 TAMS 포인터가 그 이후 할당된 모든 Old 객체를 살아 있다고 보장한다. 잘못 살리는 floating garbage는 다음 사이클에서 회수되므로 안전성을 망치지 않는다."

핵심 장치 ①

SATB Buffer + Pre-Write Barrier

"참조가 끊겨도 끊기기 직전의 객체를 살린다"

핵심 장치 ②

TAMS Pointer

"Concurrent Start 이후 새로 할당된 모든 Old 객체는 무조건 살아 있다고 본다"

SATB 철학 한 문장

"Concurrent Start 시점의 살아 있는 Old 객체는, 이 사이클이 끝날 때까지 모두 살아 있다고 본다."

이전 부 키워드 복습

키워드의미
Stop-The-World (STW)GC 동작 중 어플리케이션 코드 정지
Serial/Parallel GCYoung/Old 모두 STW → 큰 힙에서 길어지는 STW가 문제
G1의 전략① Young은 그대로 두자 (어차피 빨리 끝남) ② Old는 리전 단위 + 미리 회수 + Mark-Copy ③ Old 마킹은 어플리케이션과 Concurrent 하게

이번 핵심 질문: "Concurrent Marking을 하면 마킹 도중에 참조가 바뀐다. 그걸 어떻게 처리할 것인가?" → 답이 SATB + Pre-Write Barrier + TAMS.
Chapter 01

Concurrent Marking은 왜 위험한가

3-Color Marking 복습과 치명적 실수 시나리오.

3-Color Marking 복습

색의미
White마킹 전 (아직 방문 안 됨). 마킹 종료 시점에도 White면 garbage로 회수됨
Gray마킹은 됐지만 아직 자식 객체 스캔 안 됨 (큐에 들어가 있는 상태)
Black마킹 + 스캔까지 완료

두 가지 실수 시나리오

Concurrent 마킹 중 어플리케이션 코드가 참조를 변경하면 두 가지 실수가 가능:

시나리오결과심각도
A. 죽은 객체를 살아 있다고 판단 floating garbage. 메모리 조금 낭비. 다음 사이클에서 회수됨 OK
B. 살아 있는 객체를 죽었다고 판단 회수 후 어플리케이션이 그 객체 읽으면 → 5xx, 이상 동작, 데이터 손상 치명적
G1의 설계 원칙: B를 무조건 막는다. A는 어느 정도 허용한다.

B가 발생하는 두 조건 (둘 다 만족해야 함)

조건 ①: Black 객체가 White 객체를 향한 새 참조를 추가
조건 ②: 동시에, 해당 White 객체로 향하던 다른 도달 경로(Gray/White를 거치는 경로)가 모두 끊김
[before] GC Root → (Gray) A → (White) X ↘ (Black) B (X와 무관) [application code 실행] GC Root → (Gray) A (White) X ↘ ↑ (Black) B ──┘ ← 새 참조 추가 ↗ (참조 끊김) 이 상태에서 마킹이 끝나면: B는 이미 Black이라 다시 스캔 안 함 → X는 끝까지 White → garbage로 판단 → 회수 → 사고
포인트: 두 조건 중 하나만 막아도 B 시나리오는 발생하지 않는다.

CMS vs G1의 선택

GC막는 조건결과
CMS GC조건 ① (Black → White 새 참조)을 추적 (Incremental Update)오버헤드 큼 → deprecated
G1 GC조건 ② (White로 향하던 기존 참조의 끊김)을 추적 (SATB)본 장의 핵심
Chapter 02

해결책 ①: SATB Buffer + Pre-Write Barrier

"참조가 끊겨도, 끊기기 직전의 객체를 SATB 버퍼에 남겨서 살린다." — G1의 핵심 트릭.

Write Barrier / Pre-Write Barrier란?

용어정의
Write Barrier객체 참조가 쓰일 때 JVM이 자동으로 실행시키는 코드 조각
Pre-Write Barrier객체 참조가 변경되기 직전에 실행되는 코드 조각
⚠️ 헷갈리지 말 것: 로컬 변수(스택)의 참조 변경 시에는 동작 안 함. 객체 필드의 참조 변경 시에만 동작.
// 이때만 동작 object.field = newObject; // ← field 쓰기 직전에 Pre-Write Barrier 실행 // 이건 무관 Object local = someRef; // 로컬 변수는 트리거 X

Pre-Write Barrier가 하는 일

beforeWrite(object, field, newRef) { oldRef = object.field // 기존 참조 객체 if (concurrentMarkingActive) { localSATBBuffer.push(oldRef) // ← 핵심: 기존 객체를 SATB 버퍼에 인큐 } // ... 그 후 실제 쓰기 진행 ... }
"참조가 끊겨도, 끊기기 직전의 객체를 SATB 버퍼에 남겨서 살린다." 이것이 G1의 핵심 트릭.

SATB Buffer 구조

┌─────────────────────────────────────────────────────┐ │ Mutator Thread 1 ─┐ │ │ ├─ Local SATB Buffer (FIFO) │ │ Mutator Thread 2 ─┤ │ │ ├─ Local SATB Buffer │ │ Mutator Thread N ─┤ │ │ ├─ Local SATB Buffer │ │ │ │ 다 차거나 Remark 시점 → │ │ │ │ ┌───────────────────────┐ │ │ │ Global SATB Buffer │ │ │ │ (버퍼 주소들의 큐) │ │ │ └───────────────────────┘ │ │ ↓ │ │ GC 스레드들이 주기적으로 DQ │ │ → 자신의 Local Mark Stack에 push │ └─────────────────────────────────────────────────────┘
데이터 구조소유자역할
Local SATB Buffer각 Mutator 스레드 1개씩Pre-Write Barrier가 기존 참조 객체 push (경합 없음)
Global SATB BufferJVM에 1개가득 찬 Local Buffer의 주소 자체가 들어감 (값 옮기기 X)
Local Mark Stack각 GC 스레드 1개씩실제 마킹 진행용 (push/pop)
Global Mark StackJVM에 1개GC 스레드 간 작업 분산용 (Fork/Join 비슷)
왜 Local Buffer를 두는가? 각 스레드가 자기 버퍼만 쓰면 race condition / synchronization 비용이 0. Global로 옮길 때만 동기화.

가득 찬 Local SATB Buffer 처리

어플리케이션 스레드가 큐 비우기 작업을 하면 본연의 코드 실행이 늦어진다. 그래서 버퍼 통째로 Global로 넘기고, 새 Local Buffer를 즉시 할당받는다. 어플리케이션의 throughput을 최대한 보호.

GC 스레드의 SATB → Mark Stack 이동 (필터링)

GC 스레드가 Global SATB Buffer에서 객체를 꺼낼 때, 3가지 조건 중 하나라도 만족하면 discard:

Discard 조건이유
① TAMS 포인터 이후에 할당된 객체어차피 살아 있다고 보장됨 (§3)
② 이미 마킹된 객체 (Mark Bitmap 비트 = 1)중복 작업 방지
③ Young 객체Young은 Concurrent Marking 대상이 아님 (Young GC가 STW로 알아서 처리)
조건 ③은 사실 ①에 의해 자연스럽게 커버됨 (§3-5 참고).
Mark Stack에 push되는 조건: Old 객체 AND TAMS 이전 할당 AND 아직 마킹 안 됨.
Chapter 03

해결책 ②: TAMS (Top At Mark Start)

SATB만으론 못 잡는 케이스 — "마킹 도중 새로 할당된 Old 객체"를 처리하는 장치.

SATB만으론 부족한 케이스 — Humongous 객체 시나리오

[Concurrent Marking 도중] - Black 객체 A가 이미 스캔 완료된 상태 - 어플리케이션이 newHugeObject() 실행 → Humongous Region 새로 할당 - A.field = newHugeObject (A가 거대 객체 새로 참조)
문제:
• 거대 객체는 바로 Old generation으로 분류 (Young Eden 거치지 않음)
• Pre-Write Barrier는 기존 참조(oldRef)를 SATB에 넣음 — 새로 생긴 객체는 추적 안 함
• A는 이미 Black → 다시 스캔 안 함 → 새 객체는 끝까지 White → 회수됨 → 사고
SATB 버퍼 + Pre-Write Barrier만으론 "마킹 도중 새로 할당된 Old 객체"를 못 잡는다. 추가 장치 필요 = TAMS.

TAMS 정의

항목내용
TAMSTop At Mark Start — Mark Start 시점의 Top 포인터
존재각 리전마다 1개씩
기본값리전의 bottom (= 비어 있는 리전의 시작)
업데이트 타이밍Concurrent Start 단계에서 Old / Humongous 리전의 현재 Top 위치로 세팅
Young 리전TAMS는 항상 bottom 고정 (절대 업데이트 안 함)

리전 내부 구조 — TAMS 위치 변화

[Old Region — Concurrent Start 직전] bottom ──────────────────────────── top │■■■■■■■■■■■■■■■■■■■■■■■░░░│ └─ 기존 객체들 ───────────┘ └─ free ↑ TAMS = bottom (디폴트) [Concurrent Start 시점 — TAMS 업데이트] bottom ──────────────────────────── top │■■■■■■■■■■■■■■■■■■■■■■■░░░│ └─ 기존 객체들 ───────────┘ └─ free ↑ TAMS (이제 top과 동일) [Concurrent Marking 도중 — 새 객체 할당] bottom ──────────────────────────── new top │■■■■■■■■■■■■■■■■■■■■■■■▓▓▓│ ← ▓ = TAMS 이후 할당 └─ 기존 객체 ───────────┘└──┘ ↑ TAMS (그대로) └─ TAMS와 top 사이 모든 객체는 살아 있다고 본다
"TAMS 이후에 할당된 객체는 무조건 살아 있다고 본다."
이유: 그 객체는 Concurrent Start 시점의 스냅샷에는 없었던 새 객체. SATB의 철학상 신경 안 쓴다. Floating garbage가 되더라도 다음 사이클에서 회수.

Young Region에서 TAMS가 bottom 고정인 이유

Young Region: bottom == TAMS ↓ 할당될 때마다 top만 올라감 ┌─────────────────────────┐ │ (모든 객체가 TAMS 이후) │ └─────────────────────────┘
결과: Young에서 할당된 모든 객체는 정의상 "TAMS 이후" → 필터링 조건 ①에 의해 SATB Buffer → Mark Stack 이동이 자동으로 차단됨.

위에서 말한 "조건 ③(Young은 무시)이 조건 ①에 흡수된다"는 뜻이 이것. Young 객체는 Young GC(STW)가 다음 사이클에 Survivor로 복사하면서 알아서 처리.

각 Phase별 핵심 동작 + 시나리오 B를 막는 케이스 트레이스.

5 Phase 핵심 동작 정리

Phase모드핵심 동작
Concurrent Start STW (Young GC 피기백) (a) GC Root → Old 객체를 Mark Stack push, Mark Bitmap 비트=1, 색=Gray
(b) Old/Humongous 리전의 top → TAMS로 복사
(c) Young GC 본연 작업 (Survivor 카피)

Remark STW Local SATB Buffer → Global SATB Buffer flush → 필터링 → Mark Stack → 남은 마킹 마무리 + 약한 참조/finalizer 처리 등

영상 예제 — 시나리오 B를 막는 케이스

초기: GC Root → 1 → 5 → 8 ↓ 2 색: [] = White, () = Gray, <> = Black
시점Mark Stack5의 자식 8
1 scan → 2, 5 push[2, 5](마킹 전)
2 scan (자식 없음)[5]—
5 scan 시작 → 8 push[8](5)→8 Gray로
Mutator: 5의 참조를 끊고 2로 변경 [8] 8 참조 끊김
(Pre-Write Barrier) oldRef=8을 SATB push [8] SATB에 8
5 scan 종료[8]<5>

Concurrent Marking 종료, Remark 진입——

포인트: 8은 참조가 끊겼음에도 이미 Mark Stack에 들어가 Black이 됐기 때문에 안전. 만약 Pre-Write Barrier 없이 5가 8을 push하기 전에 참조가 끊겼다면 8은 White로 남았겠지만, SATB가 oldRef를 보존해 결국 살린다.

SATB 철학 + Floating Garbage 트레이드오프

두 가지 해결 축의 정리와 정당한 교환.

두 축 정리

축장치처리 대상
마킹 도중 참조가 끊기는 경우SATB Buffer + Pre-Write Barrier"원래 살아 있던 객체"가 끊겨도 일단 살린다
마킹 도중 새 Old 객체가 할당되는 경우TAMS Pointer"스냅샷 이후에 생긴 새 객체"는 모두 살린다

"마킹 도중 새 Old 객체 할당"의 두 케이스

Humongous 객체 직접 할당

Old 리전에 바로 할당 (Young Eden 거치지 않음). TAMS 이후 영역에 들어가서 자동으로 "살아 있음" 인식.

Promotion

Young에서 충분히 늙은 객체가 Old로 승격. 생성은 옛날이지만 "Old에 할당된" 시점이 마킹 도중. 마찬가지로 TAMS 이후.

Floating Garbage 메커니즘

GC Root → A → B (Old) A.field = null → Pre-Write Barrier가 B를 SATB push → Remark에서 B는 산다 → 이번 사이클 회수 안 됨 (Floating Garbage) → 다음 사이클: A→B 참조 이미 끊긴 상태 → SATB 안 거치고 자연스럽게 회수
트레이드오프: "살았는데 죽었다고 판단"(치명적)을 막는 대가로 "죽었는데 살았다"(메모리 약간 낭비, 다음 사이클에 정리)를 일부 허용. 정당한 교환.

Mark Stack 디테일

GC 스레드 간 work stealing과 Mark Bitmap.

Mark Stack 상태별 동작

상태동작
내 Local Mark Stack 가득 참일부를 Global Mark Stack으로 옮김
내 Local Mark Stack 비어 있음Global Mark Stack에서 일부 가져옴
Global Mark Stack도 비어 있음다른 GC 스레드의 Local Mark Stack에서 work stealing (Fork/Join Pool 유사)

Mark Bitmap

객체 주소 → 비트 위치 매핑. Mark Stack push 순간 비트=1. 영상은 1객체=1비트로 단순화 표현 (실제는 객체 정렬 단위 기준).

Mark Stack에 push되는 조건: Old 객체 AND TAMS 이전 할당 AND 아직 마킹 안 됨 (Mark Bitmap 비트 = 0) → push 후 Mark Bitmap 비트 = 1 세팅

거대 객체가 Old로 바로 가는 이유?단일 리전 크기의 50%를 초과하면 Humongous로 판정. Young Eden을 거쳐 Survivor로 카피하는 비용이 커서 처음부터 Humongous Region(Old 영역에 속함)에 할당.
CMS와 G1의 차이?CMS는 Black→White 새 참조를 추적 (Incremental Update, 비용 큼, deprecated). G1은 기존 참조의 끊김을 추적 (SATB).
Pre-Write Barrier가 로컬 변수에도?No. obj.field = x 처럼 객체 필드 쓰기에만 트리거. Object local = x는 무관.
왜 Pre-Write이고 Post-Write가 아닌가?oldRef를 캡처해야 하므로 변경 직전에. Post였다면 oldRef가 이미 사라진 뒤라 늦음.
SATB와 Remembered Set의 차이?SATB는 Concurrent Marking 중 참조 변경 안전성. RSet은 리전 간 참조 추적 (Old→Young, Old→Old). 10부에서 다룸.
TAMS는 reset 되나?각 Concurrent Start 단계마다 다시 업데이트.

면접 대비 핵심 명제

G1의 Concurrent Marking은 SATB(Snapshot At The Beginning) 알고리즘 기반.
Pre-Write Barrier: 객체 필드 참조가 변경되기 직전에 기존 참조 객체를 SATB 버퍼에 push.
SATB 버퍼는 스레드별 Local + JVM당 Global 구조. Local은 경합 없음, Global flush 시에만 동기화.
TAMS: Concurrent Start 시점 각 Old/Humongous 리전의 top 위치. 그 이후 할당 객체는 모두 살아 있다고 본다.
Young 리전의 TAMS는 항상 bottom 고정 (Young은 Concurrent Marking 관심사가 아님).
G1은 "살았는데 죽었다"(B)를 무조건 막고, "죽었는데 살았다"(Floating Garbage)는 일부 허용. 다음 사이클에서 회수.
CMS는 Black→White 새 참조를 추적 (Incremental Update, 오버헤드 큼, deprecated). G1은 기존 참조의 끊김을 추적 (SATB).
