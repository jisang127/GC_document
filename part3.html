Chapter 01

GC가 왜 등장했나

C++의 수동 메모리 관리 고통.

C++ 방식 (수동)

void allocate() { int* p = new int(10); // ... 사용 ... delete p; // 깜빡하면? }

delete 누락 → 힙 객체 영원히 남음 = 메모리 누수

Java 방식 (GC)

static void allocate() { Object obj = new Object(); // ... 사용 ... } // GC가 나중에 수거

개발자는 해제 신경 안 씀. 단 GC 검사 비용 발생.

GC 오버헤드: "이 객체가 살아 있나?"를 검사하는 비용. 이 오버헤드를 줄이는 것이 GC를 공부하는 이유.
Chapter 02

GC의 역할

OS로부터 메모리 확보·반납까지 4가지 역할.

GC가 하는 일 4가지

1OS로부터 메모리 영역 확보 (필요할 때마다 점차 늘림)
2확보된 영역 안에서 어플리케이션의 객체 생성을 받쳐줌 (미리 뭉텅이로 확보)
3사용 안 되는 객체 회수 + 콤팩션 (연속된 free 공간 확보)
4사용 안 하는 영역은 OS에 반납 (다른 프로세스가 쓸 수 있게)
Chapter 03

GC 주요 테크닉

3가지 핵심 테크닉

① 세대별 청소 (Generational Collection)

힙을 Young/Old로 나눔. 대부분 객체는 빨리 죽는다는 경험칙. 매번 전체 검사 대신 객체가 많이 죽는 Young에 집중.

② 핫스팟 집중 (Focus on densely populated area)

객체가 많이 몰린 영역만 핀셋처럼 집어서 GC 수행. 전체 무차별 검사 대신 효율 좋은 곳만.

③ 병렬화

GC 스레드를 여러 개 띄워 GC 자체를 빨리 끝내거나, 일부 GC 작업을 앱 코드와 동시 수행(Concurrent).

Chapter 05

암달의 법칙

GC가 순차 처리 부분이면 코어를 늘려도 전체 성능이 제한된다.

암달의 법칙 (Amdahl's Law)

어떤 문제에서 병렬을 통한 성능 향상은 해당 문제의 순차적인 부분에 의해 제한된다.
프로그램 전체: [──────────병렬화 가능──────────][순차] ↑ 이 부분이 병목 병렬화 가능한 부분을 아무리 잘 쪼개도, 순차 영역 길이가 전체 성능 향상의 상한이 된다.
대규모 시스템에서는 GC 오버헤드를 조금만 줄여도 캐시·Redis 같은 다른 최적화 못지않게 큰 스루풋 향상. 이게 GC를 깊이 배우는 핵심 이유.

Chapter 08

두 가지 목표 옵션

Pause Time vs Throughput — 서로 반대 방향으로 움직인다.

① MaxGCPauseMillis

-XX:MaxGCPauseMillis=500

STW 시간 상한 목표. 초과 시 힙 ↓ (객체 수 ↓ → 1회 GC 시간 ↓)

② GCTimeRatio

-XX:GCTimeRatio=99 → GC 1%

스루풋 목표. 비율 = 1/(1+N). 미달 시 힙 ↑ (GC 빈도 ↓ → 앱 시간 비율 ↑)

⚠️ 핵심 함정: 10초짜리 GC를 반으로 나눠 5초씩 두 번 해도 합이 10초가 아님. GC 한 번마다 고정 비용(STW 진입/종료, 루트 스캔, 동기화 등) 때문에 7초씩 두 번 = 14초. 빈도가 늘면 누적 GC 시간이 늘어난다.
Chapter 09

-Xms / -Xmx

초기 힙과 최대 힙 설정. Reserve와 Commit의 차이.

Reserve vs Commit

의미RAM 관계
Reserve (예약)가상 주소 공간을 연속된 영역으로 찜무관 — RAM 할당 X
Commit (커밋)그 영역에 읽기/쓰기 자격 부여매핑 가능 상태 (즉시 할당은 X)
실제 RAM 매핑데이터 읽고/쓸 때 페이지 폴트 시점이때 처음 프레임 단위로 할당
Chapter 10

서버 어플리케이션의
반전 — -Xms = -Xmx

백엔드 서버 가이드라인

java -Xms4g -Xmx4g -jar app.jar
백엔드 서버는 항상 -Xms와 -Xmx를 같은 값으로 설정한다.

왜 같게 잡나? (면접 포인트)

이유설명
자원 데디케이션서버 머신의 메모리를 JVM이 거의 다 쓴다. 다른 프로세스와 나눠 쓸 일 없음
운영 예측성힙이 늘었다 줄었다 하면 실제 힙 사이즈 예측이 어렵다
GC Pause Time 감소확장/축소 자체도 JVM이 CPU를 쓰는 일. 같게 두면 이 비용 사라짐
OOM 예방나중에 추가 commit하다 OS가 거절할 위험 제거
결과적으로 Ergonomics의 동적 조정이 사실상 무력화됨. 백엔드 개발자는 개념만 잡고, 실전에서는 -Xms = -Xmx 잘 잡는 것이 80%.
