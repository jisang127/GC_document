Chapter 01

JVM은 왜
등장했나

Platform Dependency 문제 — 환경마다 따로 컴파일해야 했던 시절.

문제 — 환경 조합마다 별도 빌드

OS / CPUIntelARM
Linux컴파일 1컴파일 2
Windows컴파일 3컴파일 4
OS마다 시스템 콜 함수 모양이 다르고, CPU마다 ISA가 다름. 기능 추가·버그 수정마다 N번씩 빌드.

해결 — Write Once, Run Anywhere (WORA)

소스코드 → [한 번만 빌드] → app.jar → 어떤 환경에서든 JVM이 실행 각 환경에 JVM만 한 번 설치하면 같은 JAR를 그대로 실행 가능.
🔑 JVM이 OS·CPU 차이를 추상화하는 중간 계층 역할을 한다.
Chapter 02

빌드 파이프라인

Compile → Package → Run 단계별 흐름.

단계별 흐름

*.java --[javac]--> *.class (바이트코드) ↓ [jar cvf app.jar -e Main *.class] ↓ app.jar (특수한 형태의 ZIP) ↓ [java -jar app.jar] ↓ JVM 프로세스 실행

주요 명령어

도구역할
javac자바 소스 → 바이트코드(.class) 컴파일
jar.class들을 .jar로 패키징 (엔트리 포인트 명시)
javaJVM 실행 파일 — JAR를 입력받아 실행
컴파일의 현대적 정의: 한 언어에서 다른 언어로의 변환. 자바 소스 → 바이트코드도 컴파일이다.

실행 파일의 정체

jar가 아니라 java가 진짜 실행 파일이다.

"JAR 파일이 실행 파일 아닌가?"

아니다. JAR는 그냥 ZIP(특수 포맷).

✓ 실제 실행 파일

$JAVA_HOME/bin/java

자바 프로세스 ≡ JVM 프로세스 (같은 말)
Chapter 05

프로세스 시작 흐름

java 실행 파일이 main 메서드까지 도달하는 과정.

JDK 소스 코드 추적

[java 실행 파일의 OS-level main()] ↓ JLI_Launch(...) ↓ CallJavaMainInNewThread(...) ↓ pthread_create(...) ← OS 스레드 생성 ↓ JavaMain(...) ← 새 OS 스레드에서 실행 시작 ├─ InitializeJVM() → CreateJavaVM() │ → 새 JavaThread(C++ 객체) + java.lang.Thread 객체 (1:1 매핑) ├─ LoadMainClass() ← Metaspace에 메인 클래스 로딩 └─ CallStaticMainMethod() ← 우리가 작성한 main 메서드 호출!
🔑 pthread_create 호출 시점에 새 OS 스레드 생성. main 메서드는 새 스레드에서 실행된다.
Chapter 06

스레드 1:1:1 매핑

자바 스레드 스케줄링은 결국 OS 커널이 한다.

3계층 매핑 구조

JAVA LAYER
java.lang.Thread
↕ 1:1
JVM LAYER
JavaThread (C++ 객체)
↕ 1:1
OS LAYER
OS Thread (pthread)
자바 스레드 스케줄링 = 결국 OS 커널이 담당. Virtual Thread는 이 매핑에서 벗어난 별개의 메커니즘.

new Thread() vs start()

호출결과
new Thread()Java 객체만 생성. OS 스레드 없음
start()OS 스레드 + JavaThread 새로 생성. 실행 가능 상태
Chapter 07

바이트코드 실행 방식
(Hybrid)

인터프리터 + JIT 컴파일러를 함께 쓰는 이유.

인터프리터 (기본)

바이트코드를 한 줄씩 해석 → 내부 C 함수 호출. CPU는 바이트코드를 모른다. JVM이 "대신 실행".

JIT 컴파일러

핫스팟 감지 → 기계어 블록 컴파일 → Code Cache 저장 → 이후 CPU 직접 실행. 성능 대폭 향상.

Tiered Compilation — 5단계

레벨동작
0Interpreter
1~3C1 (Client Compiler) — 빠른 컴파일, 낮은 최적화
4C2 (Server Compiler) — 느린 컴파일, 최고 수준 최적화
Chapter 08

클래스 로딩 &
Metaspace

필요한 시점에만 올리는 Lazy Loading과 Warm-up의 필요성.

Lazy Loading 흐름

바이트코드 실행 중 → "ArrayList 처음 등장" → Metaspace에 있나? → NO → lib/modules에서 ArrayList 클래스 정보 읽어옴 → Klass 객체 생성 → Metaspace에 저장 → java.lang.Class 객체도 함께 생성 (JVM Heap에) → 다음 호출부터는 Metaspace에서 즉시 사용

로딩 시 만들어지는 두 객체 (1:1 매핑)

객체위치용도
KlassMetaspace필드/메서드/상속/바이트코드 등 메타정보 (C++ 객체)
java.lang.ClassJVM Heap자바 코드에서 Shop.class / 리플렉션으로 접근
⚠️ Warm-up 필요: 새로 띄운 JVM은 클래스가 없고 JIT도 미완료 상태. 트래픽이 갑자기 몰리면 SSD까지 내려가서 클래스 로딩 → 느림. 배포 후 웜업 후 트래픽 투입이 표준 운영 기법.

JVM도 OS 입장에서는 그냥 프로세스 하나. 1부 개념이 그대로 적용된다.

주요 영역

영역위치저장 내용
JVM HeapJVM 내부자바 객체·배열·java.lang.Thread·java.lang.Class → GC 대상!
MetaspaceNative Memory클래스 메타정보 + Klass 객체 (JVM Spec: Method Area)
Code CacheNative MemoryJIT 컴파일된 기계어 블록
