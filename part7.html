GC 7부

Parallel GC
(Throughput Collector)

Serial GC 구조에 멀티코어 병렬성을 입혔다. 결정적 차이는 메모리 레이아웃 역전, PLAB, Ergonomics의 행동 기반 튜닝.

면접 포인트: "Parallel GC도 어플리케이션 코드와 GC 스레드가 동시에 실행되는가?" → NO. Serial이든 Parallel이든 GC 타임에는 모든 코어를 GC 스레드가 점유한다. concurrent execution이 들어오는 건 G1 GC부터다.

멀티스레드 GC

모든 코어에 GC 스레드 동시 실행. 앱은 여전히 STW.

PLAB

GC 스레드 전용 할당 버퍼. TLAB의 GC 버전.

Behavioral Tuning

목표만 설정하면 Ergonomics가 힙 크기를 자동 조정.

Chapter 02

GC 스레드 다수 운영

Serial vs Parallel 비교

Serial GC: GC time → [ GC thread 1 ] 나머지 코어는 idle singleton Parallel GC: GC time → [ GC1 ][ GC2 ][ GC3 ][ GC4 ] 모든 코어에 GC 스레드 동시 실행 어플리케이션 스레드는 모두 정지 (STW)

머신 스펙별 성능 특성

머신Parallel vs Serial
싱글 코어오히려 Parallel이 손해. PLAB/work-stealing 등 동기화 코드 경로가 그대로 작동해 오버헤드만 추가
듀얼 코어 + 중대형 힙Parallel이 약간 우세
멀티코어 (4+)Parallel 명백히 우세, 코어 늘수록 격차 커짐
Chapter 03

GC 스레드 수 공식

Ergonomics 디폴트 — n = 가용 CPU 수.

공식

n ≤ 8 → GC 스레드 수 = n n > 8 → GC 스레드 수 = 8 + (n - 8) × 5/8 (소수점 버림)

예시 계산

n (코어/HW스레드)GC 스레드 수계산
11—
44—
88—
12108 + (4 × 5/8) = 8 + 2
16138 + (8 × 5/8) = 8 + 5
32238 + (24 × 5/8) = 8 + 15
왜 5/8로 떨어뜨리나? → 스레드 간 동기화 오버헤드가 늘기 때문. 코어 수만큼 무한정 늘리면 오히려 손해. -XX:ParallelGCThreads=N으로 직접 명시 가능.
Chapter 04

메모리 레이아웃이
뒤집힌 이유

Serial vs Parallel 레이아웃

가상 주소 공간 낮은 주소 ────────────► 높은 주소 Serial GC: [ Young (Eden+S0+S1) ][ Old ] Parallel GC: [ Old ][ Young ] └ 시작 시점에 boundary 고정 (PIN) ┘

왜 뒤집혔나? — UseAdaptiveGCBoundary

문제: Serial 레이아웃[Young|Old]으로는 boundary를 못 옮김 — Old 객체들이 바로 옆 경계에 붙어있어서
조건: Serial의 '왼쪽부터 차곡차곡 콤팩트' 알고리즘을 재사용해야 함
해결: [Old(왼쪽부터 채움)][↔경계이동↔][Young] → Old는 Serial 알고리즘 그대로 재사용 가능
UseAdaptiveGCBoundary는 JDK 16에서 expired (에러). 하지만 레이아웃은 그대로 남았다. JDK 25: "The generation boundary is fixed at startup."

세 가지 목표
(우선순위 순)

목표 옵션

우선순위목표VM 옵션디폴트
1순위Max GC Pause-XX:MaxGCPauseMillis=Nno limit
2순위Throughput-XX:GCTimeRatio=N99 → GC 1% 목표
3순위Footprint(직접 옵션 없음)위 둘 충족 시 자동 축소
MaxGCPauseMillis 디폴트가 no limit이라서 사실상 Throughput Collector가 되는 것.

목표 간 트레이드오프

상황힙결과
Pause 목표 미달↓ 축소 (5%씩)1회 GC 짧아짐. Throughput은 떨어질 수 있음
Throughput 목표 미달↑ 확장 (20%씩, Young/Old 동시)GC 빈도 감소. 1회 STW는 길어질 수 있음
⚠️ Max Pause를 너무 빡빡하게 잡으면 힙이 계속 줄어 GC 너무 자주 발생 → 누적 GC 타임이 오히려 늘어남. "STW 짧게 = 무조건 좋다"가 아님.
Chapter 08

Supplement
초반 부스터

동작 원리

초기: 증가율 = Increment(20%) + Supplement(80%) = 100% 시간 지남: Supplement는 n번 GC마다 절반씩 감쇠 (>> 1, 비트 시프트) 결국: Supplement → 0, Increment(20%)만 남음
소스 코드 실제 동작:
if ((collection_count % decay) == 0) supplement = supplement >> 1;
→ n번 컬렉션마다 supplement를 절반으로. "supplement에서 8을 뺀다" / "8로 나눈다" 등은 틀린 설명.

Young GC 횟수별 총 증가율

Young GC 횟수Supplement총 증가율
0~780%100%
8~1540%60%
16~2320%40%
24~3110%30%
결국020% (Increment만)
Chapter 09

초기 힙 사이즈 디폴트

Parallel GC 초기 힙 사이즈

InitialHeapSize = 머신 메모리 / 64 MaxHeapSize = 머신 메모리 / 4 MaxYoungGen = MaxHeapSize × 1/3
Max Pause + Throughput 둘 다 충족하는 동안엔 힙이 조금씩 감소 → 안 쓰는 메모리는 OS에 uncommit → 같은 머신의 다른 프로세스 가용.
Chapter 10

Generation 사이즈
고정 vs Adaptive

사이즈를 고정시키는 조합

조합결과
-Xms == -Xmx전체 힙 고정. Young/Old 비율은 ergonomics가 조정 가능
-XX:NewSize == -XX:MaxNewSizeYoung 고정. Old만 ergonomics가 조정
-Xms == -Xmx 그리고 NewSize == MaxNewSizeYoung/Old 모두 고정 → adaptive 사실상 무력화
-Xmn NNewSize == MaxNewSize == N과 동일
⚠️ 배치 워크로드의 역설: Parallel GC를 선택한 이유 = behavioral tuning 위임. 그런데 Xms=Xmx로 고정하면 adaptive 핵심 기능 무력화. Parallel을 선택한 의미가 사라짐. 배치 워크로드에서는 Xms < Xmx로 두는 게 오히려 맞을 수 있다. 서버 어플리케이션과는 다른 의사결정 라인.
