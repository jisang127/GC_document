Chapter 00

한 줄 요약 & 4대 GC 위치

ZGC = "레이턴시에 몰빵한 GC". 힙 사이즈와 무관하게 Stop-the-World가 항상 1ms 이하인 컬렉터. 마킹뿐 아니라 리로케이션(객체 이동)까지 컨커런트로 처리한다. 튜닝 옵션은 G1보다 훨씬 적고, 대부분 디폴트로 알아서 동작한다.
면접 포인트: "ZGC의 가장 큰 특징은 힙 사이즈에 관계없이 Stop-the-World가 1ms 이하라는 것. G1GC가 마킹은 concurrent지만 evacuation은 STW였던 반면, ZGC는 relocation까지 concurrent로 처리한다."

Chapter 01

ZGC 동작 원리

G1GC vs ZGC 비교

[기존 G1GC] 어플 ▓▓▓▓▓▓▓▓▓▓▓ ▓▓▓▓▓ ▓▓▓▓▓▓▓▓▓ GC ----marking---█████----------- ← evacuation은 STW(█) (STW) [ZGC] 어플 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ GC ──marking──┃──relocation──┃─ ← 거의 다 concurrent ┃<1ms ┃<1ms STW는 아주 잠깐만
"마킹과 리로케이션 모두 컨커런트하게 진행한다. ZGC는 거의 STW가 없고, 아주 잠깐 STW만 했다가 다시 컨커런트로 진행하는 식으로 동작한다."

핵심 스펙

공식 커버 범위

수백 MB ~ 16TB 힙. 이 모든 구간에서 STW < 1ms 보장

트레이드오프

상대적으로 스루풋은 약간 떨어짐

면접 포인트: "ZGC의 STW 1ms 이하 보장은 힙 사이즈가 16TB든 1GB든 동일하다."
Chapter 02

ZGC 발전 타임라인 (꼭 외울 것)

JDK 버전변화
JDK 11ZGC가 실험(experimental) 모드로 처음 추가. Non-generational
JDK 15Production-ready 선언. 실험 딱지 떼고 옵션만 켜면 사용 가능
JDK 21 (LTS)Generational ZGC 도입. Non-gen / Gen 두 모드 공존
JDK 23Generational ZGC가 디폴트. Non-gen은 deprecated
JDK 24Non-generational ZGC 완전 제거, 관련 옵션도 사라짐

옵션 변천사

# JDK 15 ~ 20: ZGC 켜기 java -XX:+UseZGC ... # JDK 21 ~ 22: Generational 쓰려면 명시 java -XX:+UseZGC -XX:+ZGenerational ... # JDK 23: Generational이 디폴트, +ZGenerational 불필요 (deprecated) java -XX:+UseZGC ... # JDK 24+: 옵션 자체가 제거됨. Generational만 존재 java -XX:+UseZGC ...
실무 시사점: JDK 24+로 올라가면 -XX:+ZGenerational 옵션은 반드시 제거해야 함. 안 그러면 unknown option 에러.
Chapter 03

핵심 튜닝 옵션 5가지

ZGC 철학: "개발자가 옵션을 거의 안 건드려도 되도록 설계". 제너레이션 크기 / GC 스레드 수 / Tenuring threshold를 내부적으로 동적 조정.

4-1. MaxHeapSize (가장 핵심)

-Xmx16g

공식 문서가 "이것 하나만 잘 만져도 된다"고 강조하는 옵션.

힙이 클수록 ZGC 성능 잘 나옴 ("다다익선"). 단, 메모리 비용도 무시 못 함 → 밸런스. GC 로그 보면서 귀납적으로 결정.

4-2. SoftMaxHeapSize (소프트 상한선)

-Xmx5g -XX:SoftMaxHeapSize=4g

ZGC가 평소엔 4GB 안에서 동작하려고 노력. 스파이크 시 5GB까지 허용.

의미 있는 조건: Xms < SoftMaxHeapSize < Xmx일 때만 효력. 백엔드처럼 Xms==Xmx면 의미 없음.

4-3. Uncommit 관련 (ZUncommit, ZUncommitDelay)

# Uncommit 끄기 (백엔드처럼 그 서버에 자바 어플만 도는 환경) -XX:-ZUncommit # 또는 암묵적으로 끄는 방법: Xms == Xmx로 힙 고정 -Xms16g -Xmx16g # Uncommit 딜레이 조정 (디폴트 300초, 너무 짧으면 commit→uncommit 반복으로 CPU 낭비) -XX:ZUncommitDelay=300

4-4. AlwaysPreTouch (레이턴시 극한 최적화)

-Xms16g -Xmx16g -XX:+AlwaysPreTouch
면접 포인트: ZGC + Xms=Xmx + AlwaysPreTouch 조합 = "레이턴시를 극단까지 짜내는 표준 레시피". G1GC 12부에서도 동일한 권장.

4-5. UseLargePages (강력 추천, 단점 거의 없음)

[16GB 힙 기준] 4KB 페이지: 16GB / 4KB = 약 400만 개 페이지 2MB 페이지: 16GB / 2MB = 약 8,000개 페이지 ↓ TLB 미스 확률 급감 → 가상-물리 변환 빨라짐 # Large Page 설정 (Linux) echo 9216 > /proc/sys/vm/nr_hugepages java -XX:+UseZGC -XX:+UseLargePages -Xmx16g ...
효과: 스루풋 ↑, 레이턴시 ↓, 시작 시간 영향 미미 — 사실상 단점이 거의 없음. ZGC뿐 아니라 G1GC에서도 동일하게 쓸 수 있는 OS 레벨 튜닝.

4-6. ConcGCThreads (이젠 거의 안 건드림)

-XX:ConcGCThreads=N

JDK 17부터 ZGC가 알아서 동적으로 조정 → 수동 설정 거의 불필요. G1GC에서 ZGC로 마이그레이션할 때 기존 옵션을 그대로 들고 오는 경우, 빼도 된다는 신호.

Chapter 04

Full GC & Q&A

Full GC가 ZGC에 있나?

결론: 없다 (전통적 의미의 Full GC는). ZGC는 STW 방식의 Full GC가 존재하지 않음.
[전통적 Full GC] 모든 어플 스레드: ████████████████████ (전부 STW) [ZGC 한계 상황] 어플 스레드 1: ▓▓▓▓████████▓▓▓▓ (해당 스레드만 stall) 어플 스레드 2: ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ (정상) 어플 스레드 3: ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ (정상)

객체가 MaxHeap까지 다 차서 더 못 할당하는 상황이 되면 → 해당 어플리케이션 스레드 하나만 stall(일시 중지). 전체 시스템을 멈추는 방식은 없음.

백엔드라면 G1 vs ZGC?

G1GC가 밸런스 잡혀서 무난하게 쓰기 좋음. 글로벌급 또는 국가급 트래픽 서비스: ZGC 적극 고려. 트래픽이 정말 글로벌급으로 많지 않다면, 비용 걱정하면서 ZGC 힙 키우기보다 G1GC로도 비벼볼 수 있다.

튜닝 미니멈 레시피 (백엔드 기준)

java -XX:+UseZGC -Xms16g -Xmx16g \ # 힙 고정 (uncommit 자연스럽게 비활성) -XX:+AlwaysPreTouch \ # 페이지 폴트 제거 -XX:+UseLargePages \ # TLB 미스 감소 ...

OS 쪽에선 nr_hugepages만 충분히 잡아두면 끝.
