Serial GC 로그 예시

[GC #1] Pause Young (Allocation Failure) DefNew: 87000K → 1500K (96000K) 7ms ↑ Young before→after(max) ↑ STW 시간 Tenured: 1118K → 1660K ← Old 살짝 늘어남 = Promotion 발생 [GC #572] Pause Full (System.gc()) Phase 1 Mark ← 살아있는 객체 표시 Phase 2 Summary ← 새 주소 계산 Phase 3 Adjust ← 참조 포인터 업데이트 Phase 4 Compact ← 실제 객체 이동 DefNew: 16000K → 0K ← Full GC 후 Young은 0 (전부 Old로 이동)

Chapter 02

객체 생성의 문제

멀티스레드 환경에서 Eden 동시 할당의 race condition.

문제 상황

[Eden 영역 — 공유 공간] ┌─────────────────────────────────────┐ │ obj1 obj2 obj3 [Eden Top →] free │ └─────────────────────────────────────┘ ↑ ↑ ↑ Thread1 Thread2 Thread3 ... 가 동시에 할당하려 함
Tomcat 스레드풀만 해도 기본 수백 개 스레드. 동시 할당 시 Eden Top 위치를 두고 경합 발생 → Race Condition.

가장 단순한 해법 = 락 → 왜 안 되나?

객체 생성은 앱 코드에서 가장 빈번한 작업. 매 할당마다 동기화를 걸면 캐시 라인 경합이 일어나 사실상 직렬화되어 병렬성을 죽인다.

자바 설계자: "이건 자바답지 못한 해법이다."
Chapter 03

TLAB 핵심 구조

핵심 아이디어

[Eden] ┌──────────┬──────────┬──────────┬─────────┐ │ TLAB1 │ TLAB2 │ TLAB3 │ free │ │(Thread1) │(Thread2) │(Thread3) │ │ └──────────┴──────────┴──────────┴─────────┘ ↑ ↑ ↑ 각 스레드는 자기 TLAB 안에서만 객체 생성 → 경합 없음
주의: ThreadLocal API와 이름만 비슷할 뿐 별개 개념. TLAB는 JVM 내부 메모리 영역 전략.

Bump-the-Pointer 할당

생성 직전: [↑ptr] 객체1 후: [obj1][↑ptr] 객체2 후: [obj1][obj2][↑ptr] 객체3 후: [obj1][obj2][obj3][↑ptr] JIT 컴파일 후엔 CPU 명령 약 10개 수준으로 끝남. 객체 생성이 느릴 거란 직관과 정반대!

Chapter 05

GC 성능 지표

4가지 지표

지표정의
Throughput전체 실행 시간 중 GC를 뺀 앱 실행 시간 비율. 객체 할당 시간도 앱 시간에 포함
LatencyGC로 인한 STW pause 시간. 앱과 병렬로 도는 Concurrent GC 구간은 포함 안 됨
Footprint어플리케이션이 점유하는 물리 메모리 양
Promptness가비지가 된 후 메모리 재사용 가능까지의 시간
추세: Throughput → Latency 중심으로 이동. G1 GC쪽으로 넘어가는 이유. Throughput 좋아도 API 응답이 60초면 사용자 경험은 망함.
Chapter 06

-Xmx / -Xms

Reserve와 Commit의 디테일.

옵션 표

단축 옵션풀 옵션의미
-Xmx<n>-XX:MaxHeapSize=<n>JVM 힙 최대 크기
-Xms<n>-XX:InitialHeapSize=<n>초기 + 최소 크기

Reserve vs Commit 핵심 구분

의미RAM 관계
Reserve (예약)가상 주소 공간을 연속된 영역으로 찜무관 — RAM 할당 X
Commit (커밋)읽기/쓰기 자격 부여매핑 가능 상태 (즉시 할당 X)
실제 RAM 매핑데이터 읽고/쓸 때 페이지 폴트 시점이때 처음 프레임 단위로 할당
Chapter 07

서버 어플리케이션
— -Xms = -Xmx

백엔드 가이드라인

java -Xms4g -Xmx4g -jar app.jar
백엔드 서버는 항상 -Xms와 -Xmx를 같은 값으로 설정한다.

이유 — 면접 포인트

✓서버 머신 메모리를 JVM이 거의 다 씀. 다른 프로세스와 나눌 일 없음
✓힙 조정 비용 제거 (확장/축소 자체도 GC 시간에 추가)
✓운영 예측성 향상, OOM 예방
결과적으로 Ergonomics의 동적 조정이 무력화됨. MaxHeapFreeRatio/MinHeapFreeRatio 옵션 무의미.
Chapter 08

Young Generation
크기 — 5부 최대 인사이트

Young이 너무 크면

Young GC 빈도 ↓
하지만 1회 GC당 STW ↑
Old가 작아짐 → Full GC ↑

Young이 너무 작으면

Young GC 너무 자주
Premature promotion → Full GC ↑

양극단 모두 Full GC를 불러온다. 어플리케이션의 객체 할당 경향성(allocation rate)을 보고 적절한 Young 크기를 찾는 것이 핵심.

옵션 우선순위

옵션의미우선순위
-XX:NewRatio=2Old : Young 비율 (default 2)가장 약함
-XX:NewSize=<n>Young 초기/최소 크기NewRatio 무시
-XX:MaxNewSize=<n>Young 최대 크기NewRatio 무시

Survivor 크기

SurvivorRatio 옵션

옵션Default의미
-XX:SurvivorRatio=88Eden : Survivor = 8 : 1 (Young = Eden : S0 : S1 = 8:1:1)

너무 크면

To 영역이 항상 비어있어야 함 → 낭비 → Young 영역 사용 불가 면적 증가

너무 작으면

살아남은 객체가 못 들어가 너무 일찍 Old로 Promotion → Old 빨리 참 → Full GC ↑

서버 어플리케이션
튜닝 가이드라인

실전 순서

1Pause Time 허용 범위 안에서 -Xmx 최대한 크게 (물리 메모리 초과 금지)
2-Xms = -Xmx로 통일
3CPU 코어가 많을수록 힙도 크게 (스레드 ↑ → TLAB ↑ → Eden ↑ 필요)
4Young Generation 크기 충분히 확보 (Young GC에서 끝내고 Full GC 안 가는 게 핵심)
5결론: 다다익선 — 메모리 많을수록 좋다
GC 튜닝 = 영원한 모니터링 + 판단력. 공식 없음. GC 로그·메모리 그래프(톱니파 패턴)·APM 툴을 보며 평소 패턴 대비 변화 감지.
