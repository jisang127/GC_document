Chapter 00

한 줄 요약

Humongous Object는 리전 절반 이상을 차지하는 거대 객체로 올드로 분류되지만 Eager Reclamation 대상이면 영GC에서도 회수 가능(전통적으로 prim 배열 한정, 최근 JDK에서 확장). Evacuation Failure는 GC 중 대피 실패(공간부족·핀드객체)로, 실패한 영리전은 즉시 올드로 승격. Full GC는 2단계 마크-스윕-콤팩트(1차는 휴몽고스 고정, 2차에서 휴몽고스도 이동)로 동작하며(JDK 10부터 parallel) 둘 다 실패하면 OOM. Collection Set Candidate Region은 (a) 컨커런트 마킹으로 가비지 많은 올드 리전 선별, (b) Evacuation Failure를 경험한 리전 두 경로로 들어간다.

G1GC 정체성: Balanced Collector

G1 = Parallel GC의 High Throughput + MaxPauseTimeGoal 높은 확률 충족
보장메커니즘
High ThroughputGC 스레드 다수 병렬 동작 + Garbage First (가비지 많은 올드 리전 우선)
MaxPauseTimeGoal이전 GC 사이클의 통계로 휴리스틱하게 다음 영 제너레이션 크기 역산
G1GC는 Real-time Collector가 아님. "높은 확률로" MaxPauseTimeGoal 충족. 예측 실패 또는 Full GC 발생 시 못 맞춤.

Chapter 02

Humongous Object (거대 객체)

정의 & 분류

리전 사이즈의 절반 이상을 차지하는 객체 = Humongous Object. 이 객체가 들어가는 리전을 Humongous Region이라 부른다.
[중요] Humongous Object는 기본적으로 Old Generation으로 분류된다. → 영GC에서는 원칙적으로 회수 대상 아님.

배치 방식

케이스 A: 한 리전에 들어가는 경우 (리전 사이즈의 50~100%) ┌──────┬──────┬──────┐ │ ... │ Hum │ ... │ ← 남는 공간 그냥 버림 (사용 안 함) └──────┴──────┴──────┘ 케이스 B: 한 리전보다 큰 경우 → 연속된 free 리전들에 할당 ┌──────┬──────┬──────┬──────┐ │ Hum │ Hum │Hum/짜│ X │ ← 짜투리 공간은 버림 └──────┴──────┴──────┴──────┘ 회수 시: 그냥 N개 리전을 한 번에 free로 돌리면 끝

IHOP 체크 타이밍

객체 유형IHOP 체크 시점
일반 객체영GC 끝나는 시점
거대 객체할당 때마다 즉시 (사이즈가 커서 메모리를 빠르게 고갈시킬 수 있음)

Humongous Object 회수 타이밍

회수 시점조건
Remark 단계일반적인 회수 경로
Full GC최후의 수단
영GC 중 Eager ReclamationPrimitive 타입 배열인 거대 객체에 한해 (최근 JDK에서 레퍼런스 배열까지 확장)
자주 헷갈리는 포인트: "영GC는 항상 영객체만 회수하나요?" → 기본은 맞지만, 프리미티브 배열인 거대 객체가 있다면 영GC에서도 회수될 수 있다가 정확한 답.
Chapter 03

Evacuation Failure (대피 실패)

정의 & 두 가지 원인

GC 도중(YGC 또는 Mixed GC), 살아있는 객체를 다른 리전으로 옮기지 못한 상황. Full GC는 해당 없음 (Mark-Sweep-Compact 방식이라 evacuation 개념 자체가 없음).
원인설명
Allocation Failure대피할 목적지(빈 리전)가 부족
Pinned ObjectJNI 등이 네이티브에서 참조 중이라 객체 위치를 옮기면 안 됨

Evacuation Failure 발생 시 처리

[STEP 1] 대피 실패한 리전은 이번 GC에서는 회수 못 함 [STEP 2] 다음 GC 때 즉시 회수되도록 → Collection Set Candidate Region에 스케줄링 [STEP 3] (핵심) 영리전이 대피 실패했다면 → 즉시 올드 리전으로 승격
왜 영리전을 올드로 승격? Collection Set Candidate Region은 본래 올드 리전만 담는 구조. 영리전이 그대로 들어가면 구조가 깨짐 → 승격해서 자연스럽게 포함.

GC Log 패턴

Full GC 발생 전 선행 GC들에서 Evacuation Failure 로그가 연달아 찍힌다. 이것이 메모리 부족의 사전 신호.
Chapter 04

G1GC에서의 Full GC

발생 조건

조건시나리오
Free 리전 고갈객체 생성 속도 > 회수 속도
거대 객체 할당 시 연속된 free 리전 부족단편화로 인한 할당 실패
(참고: G1 Full GC는 JDK 10(JEP 307)부터 parallel. 그 이전에는 single-thread Serial Old였다.)

거대 객체 할당 실패 시나리오 (단편화) — 2차 Full GC

[할당 전] ┌──────┬──────┬──────┬──────┬──────┐ │ free │ X │ free │ X │ free │ ← 연속된 free가 없음 → 1차 Full GC └──────┴──────┴──────┴──────┴──────┘ [1차 Full GC] — 거대 객체는 그대로 두고 나머지만 Compact ┌──────┬──────┬──────┬──────┬──────┐ │X X X │ H │ free │ H │ free │ ← 휴몽고스는 위치 안 바뀜 └──────┴──────┴──────┴──────┴──────┘ ↓ 아직도 연속 free 부족하면 [2차 Full GC] — 휴몽고스도 이동 (같은 STW 안에서) ┌──────┬──────┬──────┬──────┬──────┐ │X X X │ H │ H │ free │ free │ ← 거대 객체 할당 성공 └──────┴──────┴──────┴──────┴──────┘ 2차에서도 실패하면? → OutOfMemoryError 💀

Free 리전 고갈 시나리오 흐름

빠른 객체 생성 → Eden 빠르게 참 → 영GC 발생 (객체들이 안 죽음) ↓ 대부분이 Survivor로 계속 evacuating (Aging X) ↓ Survivor도 못 버팀 → 조기 승격(Premature Promotion)으로 Old로 ↓ Old가 빨리 차오름 + 거대 객체까지 가세 ↓ IHOP 임계치 초과 → Concurrent Start 시작 ↓ Concurrent Marking 중에도 객체 폭주 → Free 리전 고갈 → Full GC
Humongous Object가 많으면 빈 짜투리 공간을 안 쓰니까 힙 고갈이 빨라진다. G1 튜닝 사례: "배열 사이즈를 줄여서 Humongous로 안 가게 했다"가 이 맥락.
Chapter 05

Collection Set (CSet)

정의 & 구성

GC를 수행하는 동안 수집 대상이 되는 리전들의 집합.

┌─────────────────────────────────────────────┐ │ Collection Set (CSet) │ ├──────────────┬──────────────┬──────────────-┤ │ Young Region │ Humongous │ Collection │ │ (Eden+Surv) │ Region │ Candidate │ │ — 항상 포함 │ — Eager 대상 │ Region │ └──────────────┴──────────────┴───────────────┘
자주 헷갈리는 포인트: "영GC의 컬렉션 셋은 영리전만이다?" → 아니다. Eager Reclaim 대상 휴몽고스 리전, Collection Candidate Region까지 포함될 수 있다.

Collection Candidate Region이 되는 두 가지 경로

경로 1

Concurrent Marking으로 선별된 가비지 많은 올드 리전 → Remark → Cleanup 단계에서 Collection Candidate Region 등록

경로 2

Evacuation Failure를 경험한 리전 → 다음 GC의 Collection Candidate Region으로 자동 등록

Chapter 06

GC Process 전체 흐름

Mixed GC 기준 흐름

[Marking 단계] [Collection 단계] SATB Algorithm Pre-Evacuation (Snapshot-At-The-Beginning) ├─ TLAB 연결 끊기 - Pre-write Barrier ├─ Collection Set 확정 - TAMS (Top-At-Mark-Start) └─ Merge Heap Roots (RemSet 통합) ↓ Evacuation (GC Root → reachable 객체 다른 리전으로 복사) ↓ Post-Evacuation (Cleanup, Mutator 재시작 준비)

영GC는 마킹 단계가 없고 GC Root에서 바로 Evacuation 시작.

GC Root가 될 수 있는 것

JNI 핸들 / Thread Stack / Static 변수 / Code Cache (JIT 컴파일된 머신코드) / Remembered Set — 엄밀히는 전통적 의미의 GC Root는 아니지만, G1에서는 Collection Set 밖에서 안으로 향한 참조를 빠르게 찾는 역할(external roots처럼 취급)

Cleanup 단계의 의미 주의: 마킹 단계의 Cleanup 페이즈(Concurrent Marking 후 단계) ≠ Collection 단계의 Cleanup(GC 전체 마무리). 같은 단어지만 다른 맥락.

면접 포인트 모음

질문답
"Humongous Object는 어디에 분류되나요?"Old Generation. 단, Eager Reclamation 대상(prim 배열, 최근 JDK는 ref 배열까지)이면 영GC 때도 회수 가능.
"Evacuation Failure 두 가지 원인?"① Allocation Failure(목적지 부족) ② Pinned Object(JNI 등으로 이동 금지)
"Evac Failure를 겪은 영리전은?"즉시 올드 리전으로 승격되어 다음 GC의 Collection Set Candidate Region이 된다.
"G1GC Full GC는 한 번에 끝나나요?"같은 STW 안에서 2단계로 동작 가능. 1차는 휴몽고스 고정+나머지 Compact, 1차로도 안 되면 2차에서 휴몽고스까지 이동. 그래도 실패하면 OOM.
"MaxPauseTimeGoal을 G1이 못 맞추는 경우?"① 통계 기반 예측 실패 ② Full GC 발생.
"Collection Candidate Region이 되는 경로?"① Concurrent Marking 후 가비지 많은 올드 선별 ② Evacuation Failure를 경험한 리전.
