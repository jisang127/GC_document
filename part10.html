Chapter 00

한 줄 요약 & 빠른 복습

G1GC Remembered Set의 역할과 세 가지 최적화 핵심.

G1GC에서 Remembered Set(RS)은 GC 루트의 추가 진입점 역할을 하며, 각 리전(또는 리전 그룹)에 대해 "외부에서 이 리전 내부를 가리키는 inbound 참조"를 기록한다. 단순 구현은 메모리를 5~10% 잡아먹기 때문에 JDK 25는 세 가지 최적화(리전 그루핑 / Rebuild RS / Merge Heap Roots)로 RS를 크게 압축한다. RS 갱신은 post-write barrier가 카드를 dirty로 마킹하고, Refinement Thread가 비동기로 dirty card를 처리해 RS에 반영한다.

핵심 명제

RS는 사실상 GC Root와 동일한 역할을 수행한다.
영GC에서 영 객체에 도달하는 추가 진입점.

Young RS (JDK 버전별)

JDK 24까지: 각 영 리전마다 개별 RS
JDK 25: 영 제너레이션 전체가 단 하나의 통합 RS 공유
이유: 영 Gen은 항상 전체가 컬렉션 대상

세 가지 최적화 한눈에

최적화타이밍무엇을 줄이나

Chapter 01

RS 필요성 & Young GC

왜 Remembered Set이 필요한가

문제: Young GC가 영 객체만 회수하려고 올드 객체 스캔을 건너뛰면, 올드→영 참조만으로 살아있는 영 객체가 "도달 불가능"으로 잘못 판단되어 회수된다.
GC Root ── B(올드) ── C(영, 살아있음) ↑ 영GC는 "올드"라며 스캔 skip ↓ C는 살아있음에도 회수됨 (BUG)
해결: 올드 객체가 영 객체를 참조하는 경우를 별도 자료구조(=RS)에 모아두고, 영GC 시 GC 루트와 함께 RS를 출발점으로 사용한다.

올드 RS에는 영→올드 참조가 안 들어간다

영 제너레이션은 매 GC마다 전체가 스캔되므로 영→올드 참조는 evacuation 중 영 객체를 따라가다 자연스럽게 발견된다. 즉 올드 RS가 추적해야 할 inbound 참조는 다른 올드 리전(CSet 밖)에서 들어오는 참조뿐.

Mixed GC & RS

Mixed GC에서 발생하는 새 문제

항목Young GCMixed GC
영 제너레이션전체 수집전체 수집
올드 제너레이션수집 안 함일부 올드 리전 선택 수집
컬렉션 셋(CSet)영 리전 전체영 리전 전체 + 선택된 올드 리전들
판정 기준이 "올드냐 영이냐"에서 "CSet에 들어있냐 아니냐"로 바뀐다. CSet에 포함된 올드 리전 → 스캔 O, 포함 안 된 올드 리전 → 스캔 X.

새로운 누락 시나리오

GC Root → A(영) → X(CSet 밖 올드) → Y(CSet 밖 올드) → C(CSet 안 올드, 살아있음) ↑ Mixed GC가 X에서 스캔 멈춤 ↓ C 회수 (BUG)
해결: 각 올드 리전은 자기 자신을 외부에서 참조하는 올드 객체를 저장하는 RS를 가진다. Mixed GC 시 RS#를 GC 루트처럼 활용해 회수 누락을 방지.

용어의미결정 시점
Collection Set (CSet)이번 GC turn에 실제로 수집할 리전들의 집합GC 시작 시 확정
Collection Set Candidate RegionsCSet 후보군 (가비지 많아서 회수하면 좋을 올드 리전 풀)Remark 단계에서 최초 확정, Cleanup에서 일부 빠질 수도 있음
Chapter 03

단순 구현의 메모리 낭비

낭비 ① — 같은 CSet 내부 참조 저장

[Region 0] ←──참조 H──── [Region 1] RS_0: {H} [Region 1] ←──참조 D──── [Region 0] RS_1: {D} 어차피 같은 Mixed GC turn에 둘 다 스캔됨 → RS_0의 H, RS_1의 D는 저장할 필요 없는 정보

낭비 ② — 중복 참조

영 RS와 올드 RS가 같은 객체(예: I)를 동시에 들고 있는 경우, 같은 CSet에 묶이면 한 번만 스캔하면 충분.

과거 GC 데이터: 단순 per-region RS만으로도 전체 힙의 5~10%를 잡아먹는 경우가 있었다. 힙이 16GB, 32GB로 커지면서 부담 가중.
Chapter 04

최적화 ① — Region Grouping

아이디어

"같은 시기에 수집될 애들끼리는 그루핑하자." CSet Candidate Regions 안에서 함께 수집되어도 되는 올드 리전들을 묶어 하나의 그룹으로 만들고, 그 그룹에 대해 RS를 하나만 둔다.

Before: After grouping: [R0] [R1] [R7] [R8] [Group A: R0, R1] [Group B: R7, R8] RS RS RS RS RS_A RS_B (4개) (2개)

제약 조건 & 효과

제약: 그룹된 리전들은 반드시 같은 Mixed GC turn에 한 번에 회수돼야 한다 (분리 수집 불가).
효과설명
메모리 절약 ①그룹 내부 참조는 저장 안 함
메모리 절약 ②그룹 내 중복 참조 자연 제거
인크리멘탈 GC와 궁합Space Reclamation Phase에서 묶어 처리 가능
Chapter 05

최적화 ② — Rebuild Remembered Set

핵심 명제

올드 리전의 RS는 Remark 페이즈 시작 후에 비로소 생성된다. 그 전까지는 안 만든다.

올드 리전은 Candidate에 선정될지조차 불확실하고, 선정되더라도 Mixed GC가 여러 번 인크리멘탈하게 도는 동안 언제 자기 차례가 올지 모른다. 매 참조 변경마다 모든 올드 리전 RS를 갱신하는 건 CPU·메모리 모두 낭비.

동작 흐름

[Remark, STW] 마킹 완료 ↓ Candidate Regions 최초 확정 ↓ [Concurrent, Remark~Cleanup 사이] Rebuild RS 실행: for each region in Candidate Regions: live 객체를 방문하면서 "이 객체가 다른 Candidate region/group을 참조하는가?" 검사 → 그렇다면 그 타겟의 RS에 현재 객체 추가 ↓ [Cleanup, STW] Candidate 파이널라이즈
중요: Rebuild는 모든 올드 리전이 아닌 Candidate Regions에 속한 리전(또는 그룹)에 대해서만 수행.

Post-Write Barrier — 컨커런트 갱신 처리

Rebuild RS는 STW 사이에 컨커런트하게 도는데, 그동안 어플리케이션이 참조를 새로 만들 수 있다.

obj.field = newRef; // 참조 변경 post_write_barrier(obj, &obj.field, newRef); // 해당 카드를 dirty 마킹

이후 Refinement Thread(-XX:G1ConcRefinementThreads)가 비동기로 dirty card를 훑어 cross-region 참조면 타겟 region의 RS에 반영한다.

Pre-write barrier(9부, SATB 유지)와 짝을 이루는 개념. Pre = 마킹 보호용, Post = RS 갱신용.
Chapter 06

최적화 ③ — Merge Heap Roots

핵심 명제

Mixed GC가 시작될 때, CSet의 모든 RS를 하나로 합친다. "Heap Root"가 RS를 의미하며, "Merge" = 합치기.

Rebuild RS로 그룹별 RS는 압축됐지만, 여러 그룹이 같은 CSet에 들어오면 그룹 간 중복은 여전히 남는다.

CSet = { Group A (R0, R1), Group B (R7, R8) } RS_A = {F, I} RS_B = {2, I} ← I가 양쪽에 중복 Merge → RS_merged = {F, I, 2} ← 중복 제거

타이밍

[Mixed GC 시작] ↓ [Merge Heap Roots] ← 여기! (evacuation 직전) ↓ [Evacuation] ↓ [Cleanup & Free]

원본 RS들은 삭제되지 않고 별도로 머지된 RS가 만들어진다.

Chapter 07

Mixed GC 전체 흐름 & Forwarding Pointer

Forwarding Pointer 메커니즘

객체를 복사할 때 원래 위치를 즉시 free하지 않고 객체 헤더에 새 주소(forwarding pointer)를 남긴다.

[Before move] [After move] ┌──────────┐ ┌──────────┐ ← 새 위치 │ Object D │ │ Object D │ │ data... │ │ data... │ └──────────┘ └──────────┘ ↑ forwarding ┌──────────┐ ← 옛 위치 │ FWD ptr │ │ →new │ └──────────┘

다른 스레드가 옛 주소로 D를 방문하면 → 헤더의 forwarding pointer 확인 → "아, 이미 옮겨졌네" → 자기 참조만 새 주소로 업데이트.

Mixed GC 종료 후 자료구조 상태

자료구조종료 후 상태
CSet에 있던 영 리전비워짐, 다음 영GC 대비
CSet에 있던 올드 리전(그룹)비워짐 + 해당 그룹의 RS도 삭제
CSet에 없던 영/올드 리전그대로 유지
영 제너레이션 RS그대로 유지 (계속 실시간 갱신)
Merged RS (임시)삭제

Q&A 헷갈리는 포인트

질문답
Young GC vs Mixed GC 판정 기준?영GC: "올드 리전인가?" → skip. Mixed GC: "CSet 안의 리전인가?" → 스캔 / 아니면 skip. 올드냐 영이냐가 아닌 CSet 멤버십이 기준.
왜 올드 RS는 영→올드 참조를 저장하지 않나?영 제너레이션은 매 GC마다 항상 전체가 스캔된다. 영→올드 참조는 영 객체를 따라가다 evacuation 과정에서 자연스럽게 발견됨.
RS에는 객체가 저장되나, 주소가 저장되나?실제로는 source 측 객체가 위치한 카드(card)의 주소가 저장된다. 카드 테이블(card table) 기반이며, GC 시 해당 카드만 정밀 스캔해 실제 참조를 찾는다.
JDK 24 vs JDK 25 영 RS 차이?~JDK 24: 영 리전별 개별 RS. JDK 25: 영 제너레이션 전체 통합 1개.
Concurrent vs Parallel 용어?Concurrent = GC 스레드와 어플리케이션 스레드가 동시에 실행. Parallel = 여러 개의 GC 스레드가 병렬 실행. GC 문서에서는 엄격히 구분.
CSet Candidate Regions는 언제 확정?Remark(STW): 최초 확정. Remark~Cleanup 사이: Rebuild RS 실행. Cleanup(STW): 파이널라이즈. Space Reclamation: Mixed GC 인크리멘탈 실행.
