GC 8부

G1GC는 어떻게
동작하는가?

Region 도입 + Garbage First 전략 + Concurrent Marking 세 축으로 Serial/Parallel GC의 두 약점을 풀었다.

핵심: "Old를 다 차기 전에 가비지 많은 일부 Region만 골라서 Young GC에 숟가락 얹어 처리한다" — 이것이 Mixed Collection이고 G1GC의 정체성이다.

Region

힙을 균등 분할. 역할이 동적으로 변함. Young/Old 경계 고정 문제 해결

Garbage First

가비지 많은 Old Region만 선택해 일부씩 컬렉션. Mixed Collection의 핵심

Concurrent Marking

Old 마킹을 앱과 동시에. STW 문제 해결. CMS의 장점 계승

Chapter 01

Serial/Parallel GC
두 약점

두 가지 약점

약점문제
① 모든 GC가 STW멀티코어를 못 살림. CPU 코어 4·8개여도 GC 동안 모든 어플 스레드 정지
② Young/Old 경계 고정프로세스 시작 시 결정. UseAdaptiveGCBoundary는 폐기됨. 트래픽 패턴이 바뀌어도 적응 불가

Chapter 02

Region
— G1GC의 새 패러다임

Region 핵심 정의

Region = JVM 힙을 균등한 크기로 쪼갠 작은 단위. 각 Region이 동적으로 Eden/Survivor/Old/Humongous 역할을 부여받는다.
┌──┬──┬──┬──┐ │E │S │O │F │ ← 2차원으로 보여도 실제는 1차원 ├──┼──┼──┼──┤ │E │O │S │H │ └──┴──┴──┴──┘ (E=Eden, S=Survivor, O=Old, F=Free, H=Humongous)

Region 크기 결정

규칙내용
목표-Xmx를 디폴트 2048개 Region으로 나눈 크기
제약1·2·4·8·16·32MB 중 2의 거듭제곱으로만 결정
불변JVM 시작 시 한 번 결정 후 불변. -XX:G1HeapRegionSize로 강제 지정 가능

Free List로 관리

Free List (linked list) top → [R1] → [R2] → [R3] → ... → [R8] ← tail 처음엔 모든 Region이 Free 상태 필요할 때마다 top에서 pop → 역할 부여 GC 후 비워진 Region은 다시 Free List로 반환 → Region의 역할은 고정되지 않고 계속 바뀐다
Humongous Object: 한 Region 크기의 절반 초과(>50%). Humongous 전용 Region 할당. 여러 Region 크기면 연속된 Free Region들을 한 번에 할당.
Chapter 03

Old GC가
더 오래 걸리는 이유

핵심 전제

GC는 가비지를 처리하는 게 아니라 살아있는 객체를 처리한다. 살아있는 객체가 많을수록 오래 걸린다. GC의 이름이 "garbage collector"라서 헷갈리지만 실은 live collector.

Old GC가 느린 3가지 이유

① 살아있는 객체가 더 많다

Young: 가비지 많음 → 살아있는 객체 적음 → 빠름. Old: 살아있는 객체 많음 (캐시·싱글톤 등) → 느림.

② 사이즈가 더 크다

Serial GC 기준 디폴트 Young:Old = 1:2. Old가 2배 큼 → 살아있는 객체 비율 같아도 절대량이 더 많음.

③ 알고리즘이 더 느리다
GCYoung 알고리즘Old 알고리즘
SerialMark-CopyMark-Sweep-Compact (compact 비쌈)
ParallelMark-Copy (병렬)Mark-Summary-Compact (여전히 compact)
CMSMark-CopyMark-Sweep (compact 없음 → 단편화 → Full GC)
Chapter 04

G1GC의 3대 아이디어

약점 → 해결책 매핑

① Garbage First — Region 선택 컬렉션

Old Generation을 통째로 컬렉션하지 말고, 가비지 많은 Old Region을 골라서 일부만 컬렉션. Old가 다 차기 전에 선제적으로. 이름의 어원: Garbage 1st.

가비지 많은 Region = 살아있는 객체 적음 = 컬렉션 빨리 끝남 ↓ Young GC에 숟가락 얹기 가능
② Evacuation — Mark-Copy를 Old에도

Old가 다 차기 전에 컬렉션하므로 Free Region이 남아있음. 살아있는 Old 객체를 Free Region으로 대피(evacuate). Compact 단계가 필요 없다 → 빠르다.

③ Concurrent Marking

Old 객체 마킹은 어플리케이션 스레드와 동시 실행 (STW 아님). CMS의 좋은 부분을 계승. CMS의 문제(단편화→Full GC 폭발)는 Evacuation으로 해결.

Chapter 06

Mixed Collection
— G1GC의 정체성

정의와 Why Piggyback

Mixed Collection = Young GC와 함께 일부 Old Region을 같이 evacuate하는 컬렉션. 영어 용어: piggybacked (숟가락 얹기).
Young GC는 자주 발생 (어차피 STW 필요)
한 번 STW 비용을 낼 거면 그 김에 가비지 많은 Old Region 몇 개도 같이 처리
→ STW 횟수와 길이 모두 최소화

Incremental 처리

Cleanup 단계에서 확정된 Collection Set 후보 = {OR1, OR2, OR3, OR4, OR5} ↓ Young GC 1회: Young 전체 + {OR1, OR2} ← 2개만 같이 처리 Young GC 2회: Young 전체 + {OR3, OR4} ← 또 2개 Young GC 3회: Young 전체 + {OR5} ← 마지막 1개

한 번에 다 처리하지 않고 incremental하게 나눠서 처리. MaxGCPauseMillis 목표를 맞추기 위해. 후보가 남아도 '회수 가치 없다'고 판단되면 중간 종료 후 Young-Only Phase로 복귀.

Chapter 07

Concurrent Marking
3단계 디테일

Stage 1: Concurrent Start (STW, Young GC에 숟가락)

GC Root ────[direct]────> Old 객체 ⟹ Mark Stack에 push GC Root ──> Young 객체 ──> Old 객체 ⟹ 무시 (간접 참조)

GC Root가 직접 참조하는 Old 객체만 push. 간접 참조 무시 → Young GC STW 최소화.

Stage 2: Root Region Scanning (Concurrent)

Young GC 끝난 직후 Survivor Region들 = "Root Region" (왜? Young GC 직후라 모두 살아있음이 보장됨) ↓ Root Region 스캔 → 직접 참조하는 Old 객체를 Mark Stack에 push

제약: 다음 Young GC 발생 전까지 끝나야 함. 못 끝내면 다음 Young GC가 대기.

Stage 3: Concurrent Marking (Concurrent)

Mark Stack에서 pop → 객체의 reference 따라가며 - Young 객체면: 무시 (관심 밖) - 이미 마킹된 Old면: 무시 - 처음 만난 Old면: 비트맵 1비트 표시 + Mark Stack에 push Mark Stack이 빌 때까지 반복. 앱은 계속 실행 중.
트라이컬러 마킹의 실체: 비트맵 1비트 + Mark Stack 멤버십. Gray(진행중) = 비트맵 1 + Stack에 있음. Black(완료) = 비트맵 1 + Stack에 없음. White(미방문) = 비트맵 0.
Chapter 08

Floating Garbage

Concurrent Marking의 약점 — 그리고 더 심각한 문제.

Floating Garbage

[T0] GC가 obj11을 살아있다고 마킹 [T1] 어플리케이션이 obj11으로 가는 참조를 끊음 (concurrent) [T2] obj11은 사실상 가비지인데 이미 "살아있음" 마킹됨 → 이번 사이클에는 살아있는 것으로 처리되어 evacuate → 다음 사이클에서야 가비지로 회수됨
G1GC가 받아들이는 트레이드오프. 이점: STW 시간 대폭 감소. 비용: 한 사이클 늦게 회수되는 가비지 존재.

더 심각한 문제 (9부 예고) — 데이터 손실!

[T0] obj A: Black (탐색 끝났다고 판단) [T1] 어플리케이션: A → C (새 참조 추가, A는 Black이라 더 안 따라감) B ✗ C (기존 Gray→White 참조 끊음) [T2] C는 사실 살아있는데 GC는 가비지로 판단 → 회수 ⟹ 데이터 손실! (치명적)
해결책 (9부 주제): Pre-write Barrier + SATB Buffer로 "Snapshot at the Beginning"을 유지.

Full GC
— G1GC의 안전망

Full GC 발동 조건

① Concurrent Marking 끝나기 전에 Old가 다 차버림
② Humongous 객체 할당 실패
③ Evacuation 자체 실패

Full GC 동작 방식

Mark-Sweep-Compact (Parallel)
완전 STW (어플리케이션 완전 정지)
매우 무거움 — 수십 초까지 갈 수도
G1GC도 Full GC 한 번 터지면 매우 아프다. 백엔드 서버에서 G1GC를 쓴다고 안심하면 안 됨. 튜닝과 모니터링이 여전히 필요. → 12부 G1GC 튜닝 가이드 참고.
Chapter 10

면접 포인트

자주 헷갈리는 포인트

Q. Mixed GC와 Concurrent Marking은 같은 건가?

아니다. Concurrent Marking = Old 객체를 살아있다고 마킹하는 페이즈. Mixed Collection = 실제로 Young + 일부 Old를 evacuate하는 단일 GC 이벤트. Concurrent Marking은 Mixed Collection을 준비하기 위해 수행됨.

Q. IHOP(Initiating Heap Occupancy Percent)는?

전체 힙 사이즈 대비 Old 점유 비율 임계치. 디폴트 45%. 이걸 넘으면 Concurrent Start 발동. Adaptive IHOP은 12부에서 자세히.

Q. GC Root → Old는 STW에서, Survivor → Old는 Concurrent로 push. 왜?

GC Root는 어플리케이션 스택 포함 → 실행 중 계속 변함 → Concurrent로 처리하면 복잡. Survivor Region은 Young GC 직후라 위치가 고정 → 다음 Young GC 전까지 안 바뀜 → Concurrent 가능.
