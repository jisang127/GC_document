Chapter 03

프로그램 vs 프로세스

같은 파일이지만 실행 여부에 따라 완전히 다른 존재가 된다.

📄 프로그램
상태정적 (실행 파일)
위치Secondary Storage
예시app.jar
⚙️ 프로세스
상태동적 (실행 중)
위치RAM에 올라간 상태
예시java PID 1234
핵심 정의: 프로세스 = 프로그램이 RAM에 올라가 실행 중인 상태
Chapter 04

컴파일 vs
인터프리팅

JVM이 인터프리팅 방식으로 동작하는 이유를 이해하는 토대.

컴파일 방식

소스코드
  ↓ [컴파일러]
실행 파일(기계어)
  ↓
프로세스

내가 작성한 코드가 직접 프로세스가 된다

인터프리팅 방식

인터프리터 (이미 컴파일됨)
  ↓ 프로세스가 됨
↓ 소스코드를 한 줄씩 읽으며
↓ 미리 컴파일된 기계어로 점프

인터프리터가 프로세스가 되어 내 코드를 해석한다

🔑 JVM 연결: JVM(Java Virtual Machine)은 인터프리팅 방식으로 동작하는 부분이 있다. JVM 자체가 C/C++로 작성된 프로그램이고, Java 바이트코드를 해석하는 인터프리터가 된다.
Chapter 05

OS의 역할

프로그램이 하드웨어를 직접 다루지 못하게 하는 이유, 그리고 OS가 중재자로서 제공하는 세 가지.

하드웨어 직접 접근의 문제

1 너무 복잡하고 어려움 — 프로그래머가 하드웨어 세부사항을 모두 알아야 함
2 시스템 전체를 망가뜨릴 위험 — 한 프로그램이 다른 프로그램 메모리를 덮어씀
3 여러 프로그램 동시 실행 시 자원 관리 불가

OS의 3대 역할

측면메커니즘
쉽게시스템 콜, 표준 라이브러리 — 복잡한 하드웨어를 추상화
효율적으로스케줄링(타임셰어링), 가상 메모리
안전하게시스템 콜로 권한 격리, 가상 메모리로 주소 격리, 동기화
Chapter 06

시스템 콜

read(0, buf, 512)를 호출하면 내부에서 실제로 어떤 일이 벌어지는가.

개념

하드웨어 접근을 추상화한 인터페이스. 함수 형태로 제공: read write mmap brk clone

OS마다 다르고, 같은 OS라도 CPU에 따라 번호가 다름. 예: Linux x86_64에서 read = 0번

read(0, buf, 512) 실행 흐름 — 클릭해서 단계별로 확인

1
인자를 레지스터에 세팅 RDI=0(stdin), RSI=buf주소, RDX=512
2
시스템 콜 번호 세팅 RAX = 0 (read의 번호)
3
syscall 명령어 실행 CPU가 특수 명령어 만남
4
커널 핸들러로 점프 유저 모드 → 커널 모드 전환
5
핸들러 실행 후 복귀 RAX 번호로 해당 핸들러 찾아 실행, 결과 반환

Privileged vs Non-Privileged

Non-PrivilegedPrivileged
누가 호출사용자 프로그램이 직접커널만 호출 가능
예사칙연산, 분기, 반복하드웨어 직접 제어
접근 방법컴파일 후 바로 실행시스템 콜을 통해서만
💡 모든 프로그램이 OS API를 호출해야 하는 건 아님 — 단순 연산(사칙연산 등)은 그대로 CPU에서 실행
Chapter 07

스케줄링과
컨텍스트 스위칭

하나의 CPU가 여러 프로그램을 "동시에" 실행하는 것처럼 보이는 비밀.

타임 셰어링 (시분할)

프로세스 A
→ 타임슬라이스 소진 →
프로세스 B
→ 타임슬라이스 소진 →
프로세스 C

CPU 코어 시간을 잘게 쪼개 여러 프로세스에 할당. 아주 빠르게 전환되기 때문에 동시에 여러 프로그램이 도는 것처럼 보인다.

컨텍스트 스위칭 오버헤드

전환 시점에 CPU 상태(레지스터, 스택 포인터, PC 등)를 메모리에 백업하고, 다음 프로세스의 상태를 복원해야 함

컨텍스트 스위칭이 너무 잦으면 그 자체가 성능 문제. 실제 연산은 안 하고 교체 작업만 반복하게 됨.

Chapter 08

RAM이 유한해서
생기는 문제들

가상 메모리가 왜 필요한지를 이해하기 위한 배경.

4가지 문제

1
프로그램이 RAM보다 큰 경우 올릴 수 없음
8GB RAM에 16GB 프로그램은 실행 불가?
2
여러 프로그램을 동시에 올리기 부족
각 프로그램이 메모리를 독점하려 함
3
External Fragmentation
빈 공간 총합은 충분해도 연속된 공간이 없어 못 올림
4
불필요한 코드까지 다 올리는 낭비
실제로는 전체 코드의 일부 기능만 사용됨
📍
Spatial Locality
어떤 주소에 접근하면 근처 주소도 곧 접근될 가능성이 높다. 배열 순회가 대표적 예.
🔁
Temporal Locality
한 번 접근한 코드/데이터를 곧 다시 접근할 가능성이 높다. 반복문이 대표적 예.
🧠 이 Locality 원리 덕분에 가상 메모리는 "자주 쓰는 것만 RAM에" 전략이 효과적으로 작동한다.

Page Fault & Demand Paging — 단계별 시뮬레이션

가상 주소에 해당하는 프레임이 페이지 테이블에 없을 때 발생하는 흐름

1
CPU가 가상 주소 접근MMU가 페이지 테이블 조회 시작
2
매핑된 프레임 없음 → Page Fault 발생해당 가상 페이지가 RAM에 없는 상태
3
커널 핸들러 실행코드 영역 → 실행 파일에서 읽어 RAM 프레임에 올림 / 스택·힙 → 빈 프레임 할당
4
페이지 테이블에 프레임 번호 기록이제 가상 주소 → 물리 주소 매핑 완성
5
CPU 재시도 → 정상 동작이번엔 페이지 테이블에 매핑이 있으므로 성공
Demand Paging: 실제로 필요해진 순간에만 RAM에 올린다. 미리 다 안 올리니까 RAM이 훨씬 효율적으로 사용됨.
Chapter 10

유저 스페이스 vs
커널 스페이스

가상 주소 공간의 두 영역

USER SPACE

코드, 데이터, 힙, 스택 — 사용자 프로그램 영역. 프로세스마다 독립적.

KERNEL SPACE

모든 프로세스에서 공통으로 RAM의 커널 코드와 매핑. 프로세스가 뜰 때 페이지 테이블에 디폴트로 설정.

🔑 시스템 콜은 사실상 같은 가상 주소 공간 안에서 커널 영역으로 점프하는 것. 다른 주소 공간으로 이동하는 게 아님!

스왑 스페이스 (Swap Space)

RAM이 다 찼는데 추가 프레임이 필요할 때의 처리 방식 3가지:

방식설명해당 영역
그냥 폐기필요하면 다시 실행 파일에서 읽으면 됨코드 영역 (read-only)
파일에 write-back변경 내용을 파일에 저장메모리 매핑된 파일
스왑으로 내림디스크 스왑 공간에 저장. 느림heap, stack 등 사용자 데이터
⚠️ 스왑은 디스크에 있어 RAM보다 훨씬 느림 → 스왑 발생량은 중요한 성능 모니터링 지표

Chapter 12

동기화

스레드 간 메모리 공유가 왜 위험한가, Race Condition이란 무엇인가.

Race Condition — a++ 예제

// 두 스레드가 동시에 a++ 수행
// 기댓값: a+2 / 실제로는 a+1 이 될 수 있음

// a++는 원자적 연산이 아님:
Thread1: LOAD a → reg1
Thread2: LOAD a → reg2 // 아직 Thread1이 저장 전!
Thread1: reg1 + 1 → STORE a // a = 1
Thread2: reg2 + 1 → STORE a // a = 1 (덮어씀!)
해결: OS가 제공하는 락(lock), 세마포어, 모니터 등 동기화 기법 사용. Java에서는 synchronized, AtomicInteger 등이 이에 해당.
Chapter 13

GC 학습으로
이어지는 연결고리

🔑 JVM도 OS 입장에서는 그저 하나의 프로세스 (대부분 C/C++로 작성). JVM도 자신만의 가상 주소 공간을 가지고, 페이지 테이블이 있고, 트루먼 쇼 속에서 동작한다.

GC를 이해하려면 필요한 OS 지식

①
가상 주소 공간 / 페이지 / 프레임

JVM Heap은 가상 주소 공간 안의 힙 영역에서 mmap/brk로 잡은 공간

②
시스템 콜 — mmap, brk

GC가 힙을 확장하거나 반환할 때 내부적으로 이 시스템 콜을 사용

③
페이지 폴트와 Demand Paging

GC가 메모리를 해제해도 OS에 즉시 반환되지 않는 이유와 연관
