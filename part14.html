GC 14부

Metaspace와
Pointer-size

GC는 힙의 객체만 회수하는가? NO. Metaspace의 클래스 메타데이터도 회수 대상이다.

면접 포인트: "GC는 힙의 객체만 회수하는가?" → NO. JVM 힙 밖 Metaspace 영역의 클래스 메타데이터도 GC가 회수한다.
토픽한 줄 요약
Metaspace 위치JVM 힙 밖의 네이티브 메모리. 클래스 메타데이터 저장
GC 트리거커밋된 양이 High Water Mark 도달 → GC 발동 (Full GC만이 아님)
클래스 회수 3조건① 인스턴스 unreachable ② 직접 참조 0 ③ 로더 자체 unreachable
UseCompressedOops일반 객체 포인터 64→32비트 (디폴트 ON). 힙 ≤ 32GB까지 커버
UseCompressedClassPointers클래스 포인터 64→32비트 (JDK 25 deprecated, 27 제거)
UseCompactObjectHeadersJDK 25 정식. 헤더 16→8바이트. 안 쓸 이유 없음
튜닝 핵심-XX:MetaspaceSize=256M -XX:MaxMetaspaceSize=512M 권장
Chapter 01

Metaspace
위치와 역할

JVM 프로세스 메모리 구조

+--------------------------------------------------------------+ | JVM Process Memory (Virtual Address Space) | | | | +-----------------------+ +--------------------------+ | | | JVM Heap | | Metaspace | | | | (Young + Old gen) | | (네이티브 메모리) | | | | - 객체 인스턴스 | | - 클래스 메타데이터 | | | +-----------------------+ +--------------------------+ | | ^ ^ | | GC 주 대상 **GC 부 대상 (가끔)** | +--------------------------------------------------------------+

핵심 속성

Metaspace는 JVM 힙 밖 — 통칭 네이티브 메모리(Native Memory)
클래스 메타데이터(타입, 필드, 메서드, 상수풀)를 저장
"메타" = 무언가를 설명하는 데이터 → 클래스 자체를 설명하는 데이터
Metaspace도 GC 대상이다. Serial/Parallel GC는 Full GC 때만 Metaspace 청크 회수. G1GC/ZGC는 Full GC가 아니더라도 concurrent하게 Metaspace 영역도 컬렉팅 가능.
Chapter 02

Metaspace
내부 구조

자료구조 계층

VirtualSpaceList (= Metaspace 전체) │ ├── VirtualSpaceNode #1 (mmap으로 reserve된 연속 주소 공간) │ ├── Chunk A1 ─┐ │ ├── Chunk A2 ─┤── ClassLoader A에 바인딩 │ └── Chunk B1 ─── ClassLoader B에 바인딩 │ ├── VirtualSpaceNode #2 (Double Linked List로 연결) │ └── Chunk C1 ─── ClassLoader C에 바인딩 │ └── ...

개념 설명

개념설명
VirtualSpaceListMetaspace 전체. 더블 링크드 리스트. 연속된 주소 공간 아닐 수 있음
VirtualSpaceNodemmap reserve된 연속된 주소 공간. 하나의 노드 내부는 항상 연속
Chunk클래스 로더 단위 할당의 최소 단위. 크기는 가변 (로더 특성/메타데이터 양에 따라)
청크는 반드시 단일 클래스 로더에 바인딩. 거꾸로: 한 클래스 로더는 여러 청크 보유 가능 (여러 노드에 걸쳐서도 OK).
Chapter 03

클래스 로더
생명주기 ★★

두 종류의 클래스 로더

종류대표 예생명주기
기본 로더Bootstrap, Application, System ClassLoaderJVM 프로세스와 동일 → 죽을 때까지 살아있음
동적 로더AOP 프록시용 로더 등다이나믹하게 생성/회수됨

클래스 메타데이터(청크) 회수의 3가지 AND 조건

[조건 1] 해당 클래스 타입의 모든 인스턴스가 unreachable AND [조건 2] 해당 클래스의 java.lang.Class 객체에 대한 참조 0 (리플렉션 보유 참조, static 멤버를 통한 도달 경로 등 모두 없음) AND [조건 3] *** 해당 클래스를 로딩한 ClassLoader 객체 자체가 unreachable *** ↓ 청크 회수 → 재사용 or OS 반환
3번 조건이 결정적. 기본 클래스 로더는 JVM과 생명주기 동일 → 절대 unreachable 안 됨 → 그 로더가 로딩한 클래스 메타데이터는 결코 회수되지 않는다.

IntelliJ Hot Reload 동작 원리

[Before Edit] [After Edit, App 살아있는 채로] ClassLoader-1 ClassLoader-1 (KILL 대상) └── UserClass v1 └── UserClass v1 ClassLoader-2 (NEW) └── UserClass v2 ↑ 이제 이걸 사용

기존 ClassLoader를 죽이고 새 ClassLoader 생성 → 바뀐 클래스를 새로 로딩. 옛 로더는 unreachable → GC가 청크 회수.

Chapter 04

Metaspace GC 트리거
— High Water Mark

-XX:MetaspaceSize — 이름 함정

⚠️ 이름만 보면 "Metaspace 초기 사이즈"처럼 보이지만 아님! 실제로는 HWM(임계치)의 초기값 = "메타스페이스가 이만큼 차면 GC를 트리거하라"는 목표치.
환경디폴트 값
OpenJDK 25 / 32비트 OS12~16MB
OpenJDK 25 / 64비트 OS16~21MB
생각보다 매우 작다. Heap이 GB 단위인 것에 비해 Metaspace는 MB 단위. 128MB 또는 256MB로 키우는 것을 권장.

HWM 동적 조정

GC 후 free 영역 비율 계산: free% = (커밋된 양 - 사용된 양) / 커밋된 양 if free% < MinMetaspaceFreeRatio (디폴트 40%) → HWM을 살짝 올림 (다음 GC 늦춤) elif free% > MaxMetaspaceFreeRatio (디폴트 70%) → HWM을 살짝 낮춤 else → 유지
Chapter 05

GC 로그 분석

Metadata GC Threshold 로그 예시

[Pause Young (Metadata GC Threshold)] ← Metaspace 이슈로 Young GC 트리거 ... Eden ... Survivor ... Old ... Metaspace: used=XX, capacity=YY, committed=ZZ, reserved=WW ← 변화 거의 없음 [Pause Full (Metadata GC Threshold)] ← 이어서 Full GC 트리거 ... Old 영역 줄어듦 ... Metaspace: used=XX, capacity=YY, committed=ZZ, reserved=WW ← 약간 증가

Metaspace 로그 용어

용어의미
used클래스 메타데이터를 실제로 저장하는 데 쓰는 크기
capacity커밋된 영역 중 청크화까지 완료되어 메타데이터 저장 가능한 크기
committedOS에 "이만큼 실제로 쓸 거다"고 요청해 mmap된 영역
reservedmmap으로 미리 예약만 해둔 가상주소 공간
로그 해석: used → capacity → committed → reserved 순으로 커진다. used가 committed에 가까워지면 곧 GC 트리거.
Chapter 06

UseCompressedOops

일반 객체 포인터를 64→32비트로 압축. 디폴트 ON.

핵심 개념

OOP = Ordinary Object Pointer (객체지향 프로그래밍 아님!)
Java 객체 주소는 항상 8바이트 정렬 (하위 3비트 = 000) 저장: [ 32비트 압축 OOP ] (하위 3비트 생략) 복원: 좌측으로 3비트 시프트 → 실제 35비트 주소 = 2^35 = 2^5 × 2^30 = 32GB 커버

커버 범위 비교

표현 방식커버 사이즈
32비트 (raw, 시프트 없음)4GB
32비트 + 3비트 시프트 (실효 35비트)32GB
⚠️ 힙 크기 32GB 초과 시 자동 OFF → 64비트 주소 사용 → 객체 크기 증가 → GC 부담↑. 31GB 정도까지만 쓰는 게 32GB로 살짝 키우는 것보다 효율적인 구간이 있다.
Chapter 07

UseCompressedClassPointers

객체 헤더의 클래스 포인터를 64→32비트로 압축. 디폴트 ON.

객체 헤더 구조

[Object Header] ┌─────────────────────┬───────────────────┐ │ Mark Word (8B) │ Class Pointer │ │ - age (GC 생존수) │ │ │ - lock state │ Compressed: 4B │ ← UseCompressedClassPointers=true │ - hash code │ Uncompressed: 8B │ ← false └─────────────────────┴───────────────────┘ 배열이면 추가: +4B (배열 길이)

헤더 크기 비교

Mark WordClass Pointer합계 (일반)합계 (배열)
8B8B (uncompressed)16B20B
8B4B (compressed)12B16B
JDK 25부터 deprecated — JDK 27에서는 옵션 자체가 삭제. 이후 무조건 압축 사용.
Chapter 08

UseCompactObjectHeaders
(JDK 25 정식 ★)

헤더 변천사

압축 옵션 없을 때: [Mark Word 8B][Class Ptr 8B] = 16B ↓ UseCompressedClassPointers (Java 8+ 디폴트 ON) Java 8~24: [Mark Word 8B][Class Ptr 4B]+pad = 12B ↓ UseCompactObjectHeaders (JDK 25 정식) Java 25+: [Mark+Class 합쳐서 8B] = 8B 배열이면: [8B] + [length 4B] = 12B

동작 방식

기존: 클래스 포인터가 주소 오프셋 방식 (위치 기반)
신규: 64비트 안에 22비트를 클래스 포인터로 사용 → 클래스 자체를 식별. 2^22 = 4M = 최대 400만 개 클래스 식별 가능

사용 방법

JDK 버전사용 방법
JDK 24-XX:+UnlockExperimentalVMOptions -XX:+UseCompactObjectHeaders (실험)
JDK 25+-XX:+UseCompactObjectHeaders (정식, 디폴트는 OFF)
이건 안 쓸 이유가 없는 옵션. 객체 크기↓ → GC 빈도↓, 캐시 히트율↑ → throughput↑. 단점 없음. JDK 25+이면 켜는 것 추천.
Chapter 09

실전 튜닝 권장값

추천 옵션 세트

# Metaspace 튜닝 -XX:MaxMetaspaceSize=512m # 무한대 디폴트 → 명시적 제한 (OOM 조기 발견) -XX:MetaspaceSize=256m # 초기 HWM (디폴트 12~21MB는 너무 작음) # Compact Headers (JDK 25+) -XX:+UseCompactObjectHeaders # 디폴트 OFF, 켜는 것 추천 # System.gc() 무력화 (라이브러리에 박혀있는 경우) -XX:+DisableExplicitGC # Compressed OOPs / Class Pointers는 디폴트 ON이라 명시 불필요

각 옵션의 이유

옵션이유
MaxMetaspaceSize=512m디폴트=무한대. 메모리 릭 시 컨테이너 OOM Killer에 죽기 전까지 계속 증가. 명시하면 OOM 조기 발견.
MetaspaceSize=256mHWM 초기값이 너무 작아 JVM 초반에 Metadata GC Threshold가 자주 발생.
UseCompactObjectHeaders단점 없고 throughput만 올라감. JDK 25+ 사용 시 켜는 것 추천.
DisableExplicitGC라이브러리 내 System.gc()는 내가 수정 불가. 옵션으로 무력화.
CompressedClassSpaceSize는 직접 튜닝 X. MaxMetaspaceSize와 묶여 자동 조정됨 (예: MaxMetaspaceSize=512m → ClassSpace도 ~420MB로 자동 축소).
Chapter 10

면접 포인트
총정리

자주 헷갈리는 포인트

Q1. GC는 힙 안의 객체만 회수하는가?

보통은 그렇지만, JVM 힙 밖의 Metaspace 영역(클래스 메타데이터)도 회수 대상. 단 3조건 AND 충족 시.

Q2. -XX:MetaspaceSize는 초기 사이즈인가?

NO. GC 트리거 임계치(HWM)의 초기값이다. 이름이 함정.

Q3. Compressed OOPs가 32GB까지만 커버하는 이유?

8바이트 정렬 덕분에 32비트 압축 OOP에 3비트 시프트 적용 = 실효 35비트 (2^35 = 32GB).

Q4. CompressedOops와 CompressedClassPointers 차이?

OOPs = 일반 객체 참조 포인터 압축 (객체→객체). ClassPointers = 객체 헤더 안의 클래스 포인터 압축.

Q5. AOP를 많이 쓰면 왜 Metaspace 이슈가?

AOP 프록시 클래스 → 동적으로 ClassLoader 생성·소멸 → 클래스 메타데이터 Metaspace 누적 → 회수 조건 만족 전까지 점유 → HWM 도달 시 GC 폭주.

Q6. Compact Object Headers는 켜야 하나?

JDK 25+이면 YES. 단점 없고 throughput만 올라간다. 디폴트가 OFF인 건 호환성 검증 단계.
