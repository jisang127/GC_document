Chapter 00

결론 먼저 & 기본 원칙

G1GC의 핵심은 "Young Generation의 크기를 유연하게 조정할 수 있어야 한다"는 것. MaxGCPauseMillis를 맞추려면 Young Gen이 어떤 컬렉션에서도 항상 수집되는 영역이라서 거기서 일감의 양을 늘리거나 줄여야 한다. 고정시키는 옵션(NewSize=MaxNewSize, -Xmn)은 절대 쓰면 안 된다. Oracle JDK 25 공식 문서 기반.

기본 튜닝 옵션 두 개

옵션의미디폴트
-XX:MaxGCPauseMillis어떤 컬렉션에서도 넘기지 않을 STW 상한200ms
-Xmx최대 힙 사이즈ergonomics가 결정 (메모리의 1/4)

스루풋 ↑ 원하면

MaxGCPauseMillis 늘리기
Xmx 크게 → GC 빈도 자체가 감소

레이턴시 ↓ 원하면

MaxGCPauseMillis 줄이기 (150ms 등)
단, 너무 타이트하면 GC 빈도↑

다른 컬렉터에서 마이그레이션 시: CMS, Parallel GC 등에서 G1GC로 넘어올 때는 기존 GC 관련 옵션은 다 지운다. 디폴트 기준으로 시작하는게 원칙.
Chapter 01

Heap 사이즈 조정

백엔드의 정석: Xmx == Xms

-Xmx4g -Xms4g # 동일하게 맞춤

힙 사이즈가 시작부터 끝까지 고정 → JVM이 OS로부터 메모리를 추가 할당/반환할 일이 없음 → 시스템 콜 오버헤드 제거.

다르게 줄 때만 의미 있는 옵션들 (참고)

옵션디폴트의미
-XX:InitialHeapSize (= -Xms)메모리 × 1/64초기 힙
-XX:MaxHeapSize (= -Xmx)메모리 × 1/4최대 힙
-XX:MinHeapFreeRatio40프리 영역이 40% 미만이면 힙 늘림
-XX:MaxHeapFreeRatio70프리 영역이 70% 초과면 힙 줄여 OS 반환 (uncommit)
리사이징 타이밍: Remark 단계 또는 Full GC 때. 단, 평가 후 OS 반환(uncommit)은 어플리케이션 코드와 concurrent하게 진행돼 STW 시간을 늘리지 않음.
Chapter 02

Young Generation 사이즈 튜닝 — G1GC의 심장

왜 Young Gen이 핵심인가?

G1GC의 모든 컬렉션(Young/Mixed)에서 항상 수집되는 영역이 Young Generation이다.
→ Young Gen의 크기를 유연하게 조정함으로써 처리해야 할 일의 양을 컨트롤 → MaxGCPauseMillis 목표 충족

권장 옵션 — 비율 기반

옵션디폴트의미
-XX:G1NewSizePercent5전체 힙 대비 Young Gen 최소 비율 (%)
-XX:G1MaxNewSizePercent60전체 힙 대비 Young Gen 최대 비율 (%)

비율 기반이라 힙 크기가 바뀌어도 그대로 작동 → 운영 편의.

절대 쓰면 안 되는 옵션 — Young Gen 고정

# 이거 절대 하지 마세요 -XX:NewSize=1g -XX:MaxNewSize=1g -Xmn1g # 위 두 개를 한 번에 설정 -XX:G1NewSizePercent=N -XX:G1MaxNewSizePercent=N # 비율이라도 동일하면 동일 결과
핵심 명제: Young Gen 크기가 고정되면 G1GC는 유연성을 완전히 잃는다. → MaxGCPauseMillis를 맞출 수가 없게 됨. G1GC에서 가장 치명적인 안티패턴.
Chapter 03

Young GC 실행 시간에 따른 튜닝

MaxGCPauseMillis를 지속적으로 초과할 때

원인: Young Gen이 이미 디폴트 최소(5%)로 줄었는데도 STW가 200ms 넘김 → 객체가 너무 안 죽음 (long-lived) 대응: G1NewSizePercent를 디폴트(5%)보다 더 줄임 (예: 3%, 1%) 힙이 64GB라면 1%만 해도 ~600MB. 작지 않은 영역이라 효과 있음.
왜 늘리는게 아니라 줄이는가?: Young Gen을 늘리면 → 더 많이 찰 때까지 기다리다가 한 번에 더 많이 살아남은 객체를 카피 → STW 더 길어짐. 방향은 줄이는게 맞음.

MaxGCPauseMillis를 간헐적으로 스파이크할 때

원인: 평소엔 OK, 가끔 Young Gen이 60%까지 부풀어 처리량 폭증 대응: G1MaxNewSizePercent를 60 → 50 등으로 줄임 → 아무리 부풀어도 상한선이 낮으니까 STW 시간도 상한선 확보
Chapter 04

Mixed GC 메커니즘 — Collection Set 3분류

Collection Set 구성

분류정체첫 단계에서 CS에 포함?
Young GenerationEden + Survivor포함
Minimum Set of Old Regions무조건 회수해야 하는 Old Regions포함
Additional Set of Old Regions시간 여유분에 추가하는 Old Regions포함
Optional Set of Old Regionsinitial pass 후 남은 시간에 incremental 처리포함 안 됨

Minimum Set 계산식

Minimum Set 크기 = (Collection Set Candidate Region 개수) / G1MixedGCCountTarget ↑ 디폴트 8 예: Remark 후 Candidate Region이 160개라면, 각 Mixed GC마다 최소 160/8 = 20개를 처리

Additional Set 동작

[Mixed GC 한 사이클] 1. Minimum Set 20개 + Young Gen 포함해서 pause time 예측 → 100ms 예상 2. 200ms - 100ms = 100ms 여유 있음 3. 그 100ms의 80% (= 80ms) 까지는 추가 회수 가능 4. Candidate에서 하나씩 빼서 누적 (가비지 많은 순으로 정렬됨, queue처럼) 5. 남은 시간이 20ms 이하가 될 때까지 추가 → 추가된 N개가 Additional Set
왜 Young GC는 200ms를 꽉 안 채우지만 Mixed GC는 꽉 채우려 하는가? → Old Region을 최대한 빨리 회수해야 Full GC를 피할 수 있어서. Mixed GC는 STW 예산을 최대한 활용.
Chapter 05

Mixed GC 실행 시간에 따른 튜닝

GC log를 보면서 판단

prediction time + eden time + base time이 표시됨. 200ms 넘는 양상을 보면 어디 문제인지 판단:

Young 쪽 (eden, base) 합이 크면 → 3장 Young GC 튜닝으로 Old 쪽 (predict initial time)이 크면 → 아래 3옵션으로

Mixed GC가 Old Region 처리에 시간 과다 사용

옵션디폴트효과
-XX:G1MixedGCCountTarget ↑ (예: 8 → 16)81회 Mixed GC당 Minimum Set이 절반 → 처리량 감소
-XX:G1MixedGCLiveThresholdPercent ↓ (예: 85 → 70)85live ratio 높은 region은 Candidate에서 제외 → garbage 많은 것만 회수
-XX:G1HeapWastePercent ↑ (예: 5 → 15)5전체 힙 대비 더 많은 % 만큼 "낭비" 허용 → tail pruning 강화
공통 아이디어: 1회 Mixed GC에 추가되는 Old Region 수를 줄이는 방향 = Old 회수가 늦어지는 트레이드오프 감수.
Chapter 06

Back-to-back GC & GCPauseIntervalMillis

문제: MMU가 0.5%

-XX:MaxGCPauseMillis = 200ms (디폴트) -XX:GCPauseIntervalMillis = 0 (디폴트, 0이면 effective = MaxGCPauseMillis + 1 = 201ms) ↑ 결과적으로 차이가 단 1ms MMU = (GCPauseInterval - MaxGCPause) / GCPauseInterval 디폴트 = (201 - 200) / 201 ≈ 0.5% → 어플리케이션이 0.5% 시간만 동작 → 사실상 멈춰 있음 (Back-to-back GC)

해결

-XX:GCPauseIntervalMillis=400 # MaxGCPauseMillis의 ~2배 정도부터 시작 → MMU 50% 보장. GC 사이 어플리케이션 코드 실행 시간 확보.
Chapter 07

IHOP & Full GC 대응

IHOP 정의

IHOP (InitiatingHeapOccupancyPercent): 전체 힙 대비 Old Region이 차지하는 비율이 이 값을 넘으면 Concurrent Start GC를 시작.
옵션디폴트의미
-XX:+G1UseAdaptiveIHOPON (true)IHOP을 동적으로 결정
-XX:InitiatingHeapOccupancyPercent45초기값/고정값
-XX:G1ReservePercent10버퍼 (0~50). Old가 (100-10)%=90% 찰 때쯤 Mixed GC 시작되게 IHOP 계산

Full GC 발생 시 대응 방법

Full GC가 났다 = 디폴트 설정만으로 감당이 안 된다는 신호. 대부분 객체 생성 속도가 너무 빨라서 Old Gen이 꽉 차버린 상황.

방법옵션비고
1. 힙 사이즈 늘림-Xmx, -XmsRAM 필요
2. Concurrent GC 스레드 증가-XX:ConcGCThreads어플리케이션 스루풋 하락
3. Reserve Percent 증가-XX:G1ReservePercent유효 힙은 줄어듦
4. Adaptive IHOP 끄고 IHOP 명시-XX:-G1UseAdaptiveIHOPAdaptive 이점 포기
5. Region 사이즈 증가-XX:G1HeapRegionSize=NHumongous 임계치 상승
System.gc() 처리: -XX:+ExplicitGCInvokesConcurrent (Full GC 대신 Concurrent하게) 또는 -XX:+DisableExplicitGC (아예 무시)
Chapter 08

GC Log의 sys 값이 클 때 — 시스템 시간 튜닝

GC log 읽는 법

[Times: user=0.42 sys=0.30 real=0.05 secs] ↑ JVM 코드의 총 합 ↑ OS 커널 코드 시간 ↑ 실제 STW 경과 시간 (병렬이라 real != user)

sys가 비정상적으로 크면 OS 호출 비용이 큰 상태.

해결책 3+1가지

(1) Xmx == Xms

힙 사이즈 변화 = OS commit/uncommit → sys 소비. 동일하게 두면 그 비용 제거.

(2) AlwaysPreTouch

-XX:+AlwaysPreTouch
JVM 시작 시 전체 힙 페이지를 한 번씩 터치해서 미리 매핑. 런타임 page fault 제거. 트레이드오프: 부팅 시간 증가.

(3) Transparent Huge Pages 끄기

OS가 페이지를 동적으로 합치는 작업이 GC pause에 악영향. 운영 시스템에서 THP 기능 끄는 게 권장.

(4) GC 로그 I/O 병목

GC 로그 write가 다른 I/O와 경합해서 STW 시간이 늘어나는 경우. 로그를 별도 스토리지에 저장하거나 비동기 로깅 사용.

Chapter 09

Q&A & 핵심 정리

자주 헷갈리는 포인트

질문답
Young GC와 Mixed GC에서 Young Gen 크기가 다르게 잡히나?예. Mixed GC가 더 작게 잡힘. Mixed GC는 Young + Old 처리 → 시간 예산을 Old에도 할애하므로 Young Gen이 더 작음.
G1UseAdaptiveIHOP을 켰는데도 IHOP 초기값이 의미 있나?Adaptive가 통계를 누적하는 데 시간이 걸림. 초기 몇 사이클 동안은 InitiatingHeapOccupancyPercent 값을 사용. 통계가 쌓이면 자동 계산.
Mixed GC도 MaxGCPauseMillis를 꽉 채우면 Back-to-Back 위험?발생 가능. 그래서 GCPauseIntervalMillis를 MaxGCPauseMillis의 2배 정도로 잡는 게 안전책.
