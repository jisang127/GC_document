GC 종류 핵심 정리
(JDK 25 기준)

처리량(Throughput) vs 지연시간(Latency) 트레이드오프가 Parallel ↔ ZGC 양 극단을 결정한다. G1은 그 사이의 균형.

Serial / Parallel

STW 방식. Serial은 싱글코어용, Parallel은 배치·스루풋용

G1 GC (디폴트)

JDK 9~25 서버 클래스 디폴트. Throughput과 Latency의 균형

ZGC / Shenandoah

초저지연. ZGC는 1ms 미만 보장. 거의 100% Concurrent

Chapter 02

Parallel vs Concurrent
— 중요 정의

Parallel

GC 스레드 N개가 모두 GC만 수행 → 어플리케이션은 무조건 멈춤 (STW)

[Parallel GC, 4코어] 코어1: [GCGCGCGCGCGCGC] 코어2: [GCGCGCGCGCGCGC] ← 전부 GC 코어3: [GCGCGCGCGCGCGC] ← 앱은 0 코어4: [GCGCGCGCGCGCGC]

Concurrent

GC 스레드가 도는 동안에도 일부 코어에서 어플리케이션 코드가 계속 실행

[Concurrent GC (G1/ZGC), 4코어] 코어1: [GC GC GC GC GC] 코어2: [GC GC GC GC GC] ← 일부는 GC 코어3: [APP APP APP APP] ← 일부는 앱 코어4: [APP APP APP APP]
핵심: Parallel GC도 멀티 스레드지만 컨커런트는 아니다. Concurrent = GC 스레드가 여러 개라는 뜻이 아님.
Chapter 03

Serial GC

핵심

싱글 스레드로 동작 — 가장 최초의 GC 형태
Generational: Young(아래) + Old(위)
언제 쓰나: 싱글 코어 환경 / 힙 100MB 이하 소규모

컨테이너 환경의 함정 (면접 단골)

호스트Docker 옵션결과
16GB / 8코어--cpus=1 -m=1gclient class → Serial GC 디폴트!
16GB / 8코어제한 없음server class → G1 GC 디폴트
컨테이너에서는 호스트 머신이 아닌 컨테이너의 CPU/메모리 크기가 기준. -XX:+UseG1GC 명시 필수.
Chapter 04

Parallel GC
(Throughput Collector)

핵심 철학 — 스루풋 최우선

스루풋 = 어플리케이션 실행 시간 / 전체 실행 시간 전체 20초 = 어플리케이션 19초 + GC 1초 → 스루풋 95% 한 번 STW가 길어도 좋다. 전체 실행 시간을 줄이는 게 중요.
⚠️ 단점: 힙 4GB 넘으면 STW가 수초~수십초까지 갈 수 있음. 배치 처리·데이터 처리 워크로드에 적합.

디폴트 GC 역사

JDK 버전디폴트 (server class)
5 ~ 8Parallel GC
9 ~ 25G1 GC
Chapter 05

G1 GC
(JDK 9~25 디폴트)

Mostly Concurrent Collector

[G1 GC의 한 사이클] ─STW─├─────CONCURRENT (앱과 동시)──────┤─STW─├──CONCURRENT──┤ 짧음 여기서 앱 코드도 함께 돌아간다 짧음
수십~수백ms 수준의 STW. 소규모(듀얼코어, 수GB)부터 대규모(32+코어, 수백GB)까지 광범위하게 커버 → 디폴트로 채택.

특징

높은 스루풋 (Parallel만큼은 아니지만 충분히 높음)
높은 확률로 MaxPauseMillis 목표 준수 — 500ms 지정 시 거의 안 넘김
Generational이지만 Region이라는 새 개념 도입 (G1 심화는 8부~12부에서)
Chapter 06

ZGC
(Ultra-Low Latency)

핵심 — 굉장한 GC

힙 사이즈와 무관하게 1ms 미만의 STW를 보장한다.
힙이 수백MB든 16TB든 — STW는 일관되게 1ms 미만. 최선이 아니라 보장.

Generational ZGC 진화

JDK동작
11ZGC 실험적 도입 (논제너레이셔널)
15ZGC production-ready
21Generational ZGC 등장 (production-ready)
23Generational이 디폴트 모드
24논제너레이셔널 완전 제거
25Generational ZGC만 존재
Generational ZGC가 비제너레이셔널보다 스루풋 10~20% 개선. 어떤 경우 G1보다도 스루풋이 잘 나온다. -XX:+UseZGC 명시 필수 (아직 디폴트 아님).
Chapter 07

CMS GC — 역사

현재는 사용 불가 (JDK 14 완전 제거). 인터뷰·문서에서 만날 수 있어 알아둘 가치는 있음.

왜 사라졌나 — Compact 단계의 부재

CMS Sweep 후의 힙 (Compact 없음): [●][gap][●][gap][●][gap][●][gap][●][...큰 free 공간...] Sweep만 하고 압축(Compact) 없음 → 단편화(Fragmentation) 발생 작은 빈 슬롯들을 Free-list로 관리 큰 객체 할당 요청 → Free-list 탐색 실패 → Full GC 발동 Full GC는 싱글 스레드로 Compact 수행 → STW가 엄청 길어짐 예측 불가능한 거대 STW → 운영 불가
"갑자기 Full GC 돌면서 STW가 수십초로 튀었다. SNS처럼 트래픽 몰리는 서비스에서는 장애로 직결."
컨커런트 컬렉터의 바통: CMS → G1 → ZGC/Shenandoah
