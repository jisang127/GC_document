Generational GC
(세대별 컬렉션)

대부분의 객체는 태어나서 금방 죽는다 — 이 경험칙이 Generational GC의 모든 것을 설명한다.

핵심 결론: GC는 이름과 달리 garbage가 아니라 live object를 추적한다 — 사실은 "Live Collector"에 가깝다.

Weak Gen. Hypothesis

대부분 객체는 금방 죽음. 오래된 객체는 젊은 객체를 거의 참조하지 않음.

Mark-Copy

Young에서 사용. Live object만 복사 → 비용이 live 수에 비례.

Remembered Set

Old→Young 참조를 별도 관리. Minor GC의 추가 GC Root.

왜 Generational인가

통짜 힙의 한계와 공간 분리 해결책.

Weak Generational Hypothesis

객체 수 ↑ ▲ │ █ │ █▓ │ █▓░ │ █▓░░ ... ▲ (오래 살아남은 객체) └──────────────────→ 라이프타임 대부분 여기서 죽음 일부는 끝까지 산다
"대부분의 객체는 태어나서 금방 죽는다" — GC의 핵심 동작 원리

공간 분리 해결책

┌──────────────┬──────────────────┐ │ Young Gen │ Old Gen │ └──────────────┴──────────────────┘

Young과 Old를 물리적으로 분리 → young만 컬렉팅 시 old는 건들지 않음 → 주소만 봐도 young/old 구분 가능.

Chapter 03

Remembered Set

Old → Young 참조를 별도로 관리하는 이유.

문제 상황

[Old object] ───참조───→ [Young object] Young만 컬렉팅 시 GC Root에서 Old 객체로 가지 않으므로, Old가 참조하는 Young 객체를 놓침 → garbage 오판!

Remembered Set 정의

항목내용
무엇을 저장Young 객체를 참조하고 있는 Old 객체의 주소
동작 시점Minor GC 때만 사용
역할Minor GC 때 추가적인 GC Root 역할
💡 면접 포인트: "리멤버드 셋은 마이너 GC에서 GC Root 역할을 한다"
Chapter 04

GC 알고리즘 3종

① Mark-Sweep — 단순하지만 단편화

처리 전: [live][free][live][free][free][live] 처리 후: [live][ ][live][ ][live] ↑ 총 free 공간은 있어도 연속 공간이 없어 큰 객체 할당 실패

② Mark-Sweep-Compact — 압축 추가, 비용 높음

4단계 (Linear Scan 3회): 1. Mark: GC Root부터 live object 마킹 2. Compute: live 객체의 새 주소 계산 → 헤더에 Forwarding Address 3. Update: 모든 참조를 새 주소로 갱신 4. Copy: 실제 객체 이동 비용: 힙 크기 + 전체 객체 수에 비례 (비싸다)

③ Mark-Copy ★ 가장 중요 (Young에서 사용)

빈 공간을 따로 잡아두고, GC Root에서 따라가며 live만 복사 [현재 영역 꽉참] → [빈 공간(예약)] [A][G][B][G][C] [ ] ↓ 복사 후 [free 영역 ] [A][B][C]
비용이 live object 수에 비례. 대부분 객체가 죽으면 live가 적음 → Mark-Copy가 Young에 잘 맞는다.
Chapter 05

Eden + Survivor 0/1
최종 구조

왜 Survivor가 두 개인가?

Mark-Copy는 빈 공간이 필수. 다음 Minor GC에서도 Mark-Copy를 쓰려면 복사 대상 공간이 필요하다.

Eden(생성용) + Survivor 0(현재 사용) + Survivor 1(다음 카피 대상) → 항상 하나는 비어있어야 한다

영역 구조와 디폴트 비율

┌───────────────────────────────────┬──────────────────┐ │ Young Generation │ Old Generation │ │ ┌──────────────┬───────┬───────┐ │ │ │ │ Eden │ S0 │ S1 │ │ 2 │ │ │ 8 │ 1 │ 1 │ │ │ │ └──────────────┴───────┴───────┘ │ │ └───────────────────────────────────┴──────────────────┘ Young : Old = 1 : 2 / Eden : S0 : S1 = 8 : 1 : 1
Survivor가 저렇게 작은 것 자체가 '대부분 객체가 금방 죽는다'는 가설을 디폴트 값이 반영하는 증거.
Chapter 06

Minor GC 동작

Mark-Copy + Forwarding Address.

Minor GC 시나리오

1트리거: Eden이 가득 차서 새 객체 할당 불가
2GC Root 탐색: GC Root + Remembered Set에 등록된 Old 객체에서 출발
3Live 객체 복사: S0(빈 Survivor)로 복사. 원래 위치 헤더에 Forwarding Address 기록
4Age 임계치 도달: S0가 아닌 Old로 Promotion. Old → Young 참조면 Remembered Set 등록
5완료: Eden + S1(이전 Survivor) 전부 free 처리. 다음 Minor GC에서 역할 스왑
Chapter 07

Major GC / Full GC

왜 Full GC가 느린가.

Full GC 트리거 조건

트리거설명
Young GC 직전 휴리스틱평소 promotion 양을 보고 Old에 공간 부족 예상 → Full GC
Promotion 실패Minor GC 중 Old에 공간 없음 → Full GC
Huge object 할당 실패큰 객체는 Old에 바로 할당, 공간 없으면 Full GC

Minor GC vs Full GC 비교

요인Minor GCFull GC
대상 공간Young만전체 힙
객체 수적음 (대부분 죽음)많음 (Old에 누적)
알고리즘Mark-Copy (live에 비례)Mark-Sweep-Compact (전체에 비례)
⟹ 단순히 공간만 넓어진 게 아니라 알고리즘 자체가 더 비싼 것을 쓴다.
Chapter 08

핵심 인사이트

GC는 사실 "Live Collector"다.

통념 vs 실제

통념실제
Garbage가 많을수록 오래 걸린다?Live object가 많을수록 오래 걸린다
알고리즘을 뜯어보면 추적 대상은 garbage가 아니라 live object다.

실무 영향: 로컬 캐시(Caffeine 등)의 함정

→캐시된 객체는 항상 reachable ⟹ GC가 매번 다 추적해야 함
→절대 수집되지 않음 (참조가 계속 유지) → age 차서 Old에 쌓임
→Full GC 빈도 ↑, GC 시간 ↑, 회수되는 메모리는 거의 없음
트래픽이 많은 백엔드에서는 "캐시를 무조건 많이 넣는 것"이 GC 성능을 갉아먹을 수 있다.
